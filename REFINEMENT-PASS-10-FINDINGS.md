## 2. The one editorial theme worth discussing: uneven steelmanning

The book promises "we weigh the naive option *and* a rival engineer's, each with genuine trade-offs." It usually delivers, superbly (ch.2's three cases where a SPA is correct; the PostgreSQL/XTDB costing in ch.8-9). But a recurring set of **load-bearing choices arrive asserted, not argued** — and a reader who knows the alternative feels the strawman. Each is a 1–3 sentence fix (restore the rival), not a rewrite. This is the highest-leverage change and the most in-character with the book's method. Per instance:

- **Ch.2 — The React "penance narrative" overreaches.** signals/fine-grained reactivity (Solid, Svelte runes) and compile-away-the-vDOM are largely about client-side update cost, not "recovering server authority" — folding them into the debt-repayment story is selective. And the section headlines *addressability* as the signature free win, which is the cost modern meta-frameworks (file routing + loaders + PE forms) have most recovered — meeting the strongest rival at its strongest point without saying so. The durable wins (no model duplication, no hydration tax, no second build) are firmer ground.
- **Ch.4 — zprint chosen; cljfmt / cljstyle never named.** formatting is asserted ("We use zprint") in a chapter that steelmans the linting gate two paragraphs earlier. The rival's obvious reach for cljfmt goes unanswered anywhere in the manuscript.
- **Ch.7 — tick waved off on the wrong axis.** dismissed as giving "types we do not need," when tick's actual clock facility is a `with-clock` over a dynamic var — i.e. exactly the binding-conveyance pattern this chapter spends a paragraph rejecting. The honest line ("tick gives a mockable clock, but via the dynamic-var conveyance problem we just described; we won't take a dep for one macro") is stronger and true.
- **Ch.10 — fresh-DB-per-test asserted as obviously right.** the standard rival — transaction-rollback-per-test / template DB — exists in SQL shops precisely because fresh-per-test is expensive there. The bridge ("the tool is what makes the cheap choice cheap") is missing; the strongest counter to the chapter's central claim is never named.
- **Ch.12 — CSS-in-JS dismissed in two sentences.** the live rival for a zero-dependency Clojure book — a small hand-authored design-token CSS file (or Garden), no npm, no scanner, no dynamic-class footgun — is never named, even though the chapter's own hand-written-CSS section half-concedes the point. Connects to a real cross-chapter tension: ch.11 rolls its own i18n to avoid a dependency, ch.12 adopts an external Node CLI — the flip in the dependency calculus is never reconciled.
- **Ch.13/14 — idiomorph adopted as a given.** the single most load-bearing front-end dependency arrives without options: morphdom / nanomorph appear nowhere. Worse, *why* idiomorph's id-set matching preserves focus/scroll/form-state (vs a naive innerHTML swap) is asserted as an outcome and never shown — the exact mechanism a working engineer wants.
- **Ch.19 — LiveView / Livewire gets a one-line strawman.** "pull ephemeral state back to the server and turn the browser into a dumb terminal" is the SSR-purist caricature; LiveView ships JS hooks and `phx-update="ignore"` for exactly the ephemeral-in-the-browser cases the chapter says it can't handle. Below the book's own steelman bar.
- **Ch.20 — passwords dismissed, not steelmanned.** "Passwords are a liability … magic links sidestep all of this," with no weighing: no password-manager ergonomics, no instant/offline login, and — for a security book — no admission that passwordless makes the email account a single point of failure with no second factor (the text even *assumes* the inbox has MFA, often false).
- **Ch.26 — Forgejo under-optioned.** chosen almost entirely on "you already self-host the forge"; rival self-hosted forges/runners (Woodpecker, Drone, GitLab CI) are never weighed — the only steelmanned alternative is hosted GitHub Actions.

*Related "confident no-downside framing where a real cost exists" (same family, one now fixed):* Ch.21 session revocation **(fixed, §1)**; Ch.24 the one file exempt from content-hashing (`idiomorph-*.min.js`) rests on an **unenforced** "version string changes when bytes do" convention + a one-year immutable cache — the exact lying-name failure the whole gate exists to prevent, never acknowledged; Ch.11 the graceful-fallback affordance and the mandated parity test contradict (using the fallback turns CI red), and in prod a missing key silently ships a raw slug to a user in a book that refuses to boot without key material.

*Related "show-don't-tell inversion" (the book shows one-liners but narrates its hardest code):* Ch.9 `line-diff` — billed as "the densest function in the namespace," it is the one function elided to a prose sketch; Ch.16-17 dossier drill-downs (`branches`, `hiccup`) are enumerated in one bullet each while `flow` gets a full worked example.

## 4. Craft & positioning (identity-adjacent — discuss, don't just "fix")

- **Density tic → fatigue, not collapse.** The prose leans hard on two devices — the em-dash parenthetical (multiple per paragraph) and the "not X; Y" antithesis. Individually elegant; in aggregate a detectable, machine-even cadence that risks wearing the reader down over 118k words. The density is *identity* and must not be sanded down; the cheap, non-betraying mitigation is rhythmic variety — modulate the two tics, and let a rare human aside or concrete war story break the airless evenness (the two things DDIA does that this book, at its best, still does not).
- **No warmth / humor / visual delight.** The book makes you respect it; it never makes you smile. Nystrom's wit and hand-drawn figures and Evans' human voice are absent by disposition. A real cost over 26 chapters (affection is what carries a long book) — but a deliberate one. Flag, don't necessarily change.
- **Title oversells one axis, undersells its true one.** All six market lenses agreed: readers hear "SaaS" as billing + multi-tenant teams (both deliberately out of scope), while the book's real, uncontested axis is *production-grade, server-authoritative, framework-free, hardened, own-your-stack*. Consider leading the positioning with the true axis and treating "SaaS" as the vehicle — otherwise it draws ship-fast buyers who feel the omissions and under-reaches the own-your-stack readers it is perfect for.
- **Audience = premium niche.** Realistic lifetime serious readership is low-thousands (Clojure ∩ Datomic ∩ no-framework ∩ dense/no-exercises). Judge it as a niche flagship and a credibility/consulting asset, not a volume seller. The Datomic coupling is simultaneously the chief differentiator and the chief deterrent, and the book already handles it about as well as possible (free Pro + `datomic:mem`, itemized Postgres/XTDB escape).

### Stateless session cookies have no server-side revocation story (leaked-cookie window, no force-logout / log-out-all-devices, no admin kill of a compromised user's live sessions)
**Verdict:** `genuine_gap`  **Severity:** worth_a_paragraph  
**Evidence:** Ch.21 presents the stateless encrypted cookie as pure upside and never names the revocation cost. 21:362 "No server-side session store... No session table to query, no cleanup jobs, no scaling concerns." The dedicated section "Why a client-held session is safe" (21:364-372) argues only integrity/forging — an edited blob fails the MAC and Ring reads no session — and says nothing about revoking a VALID cookie. 30-day expiry is a bare bullet (21:360). Logout is `(assoc :session nil)` with "That is the entire logout implementation" (21:449-456) and no note that it clears only the requesting browser's cookie; there is no log-out-all-devices, no admin ability to kill a compromised user's live sessions, and no step-up re-auth on the same-cookie admin path (Ch.23). This omission is conspicuous because the book names every OTHER security cost of its simplifications: CSRF-token absence (21:376), login-CSRF (21:380), the mutating-GET verify (21:473), the single-instance rate limiter (21:187), and the nonce-store-wipe replay caveat (21:189). It even bounds the 15-min magic-link *token* leak and builds one-shot nonce consumption (20:193, 21:189, 21:251) — yet the far longer-lived *session* the token mints has no equivalent revocation story. Not in the afterword's explicit out-of-scope list (afterword:19 concedes horizontal scale, billing, observability, backup, background jobs, email service, offline sync — not session revocation).  
**Recommendation:** Add one short honest-caveat paragraph in the "Why a client-held session is safe"/Logout region of Ch.21, parallel to the existing CSRF caveat: state that the cost of a self-authenticating cookie is that a session cannot be revoked server-side — a stolen/leaked cookie is valid for its FIXED 30-day life (do NOT write "sliding"; see soundness note), logout clears only the current browser, there is no log-out-all-devices, and the only global lever is rotating SESSION_KEY, which signs everyone out. Then give the natural fix path so it's a named trade rather than a hole: because `wrap-current-user` already resolves the user from the DB every request, add a `:user/sessions-valid-after` instant (or a `:user/disabled?` flag) on the user entity plus an `:issued-at` in the session map, and reject in that existing pull when the cookie predates the stamp — "log out all devices" becomes one timestamp write, admin-disable becomes one flag flip. Keep it to a paragraph; a recipe box is a low-value target and the book's own "name the gap, site the fix" idiom (as at 21:380) fits exactly.  
**Soundness check:** Verified the core claim and the proposed fix against the repo and the Ring source. (1) Revocation gap is real: session content is only `{:user-email ...}` (handler.clj:235); logout is `(assoc :session nil)` (handler.clj:259) affecting one browser; nothing checks a per-user validity stamp. (2) IMPORTANT correction to the candidate's framing: its supporting subclaim that "Ring re-sets the cookie each response ⇒ sliding 30-day window that never expires" is technically WRONG for ring/ring-core 1.15.3 (deps.edn:6). `ring.middleware.session/bare-session-response` computes a new cookie only `when (contains? response :session)`, and re-emits Set-Cookie only when the new key differs or per-response attrs are set — so the cookie is refreshed ONLY on responses where a handler explicitly writes `:session` (here: login and logout). Ordinary authenticated reads never touch `:session`; `wrap-locale` (i18n.md:200-206) reads `:session` but never writes it and nothing ever sets `:session :locale`. So max-age is fixed at login (30 days), not sliding — added text must say "fixed." (3) The proposed fix is sound and cheap: `wrap-current-user` (routes.clj:70-71) already does a per-request email→eid DB read, so adding `:user/sessions-valid-after`/`issued-at` comparison is near-free and is the standard iat/token-version invalidation technique; note also that deleting the user entity ALREADY invalidates their sessions via wrap-auth's `(assoc :session nil)` (routes.clj:95-96), which is a blunt account-deletion lever, not per-session revocation.

### Connection lifecycle: why per-request d/connect is safe (peer connection caching), and behavior on transactor/storage disruption

**Evidence:** chapters/08-datomic.md:62-63 defines `get-connection` as `(d/connect (db-uri))`, and it is invoked per-request across production code (src/myapp/web/handler.clj ~15 sites; routes.clj:71,107; chapters/13-hiccup-views.md:362; chapters/19-progressive-enhancement.md:297) with no pool and no d/release. ## 6. Market comparison (verified, condensed)

### Clojure book landscape
**Wins:** • It fills a genuinely empty slot: no current (2024-2026) book builds a real Clojure web app from an empty directory with no client framework, server-rendered, and production-hardened. The only book-market competitor for 'build a Clojure web app,' WdC 3rd ed, is SPA/re-frame-first, framework-assisted (Luminus), relational-DB, and dated to 2021. • Datomic is a near-monopoly: no in-print book treats Datomic as the spine. The last one to touch it seriously (Professional Clojure, 2016) is a decade stale and gave it one chapter. This book makes Datomic's time-travel (d/history, d/as-of) the product itself. • It is on the right side of history for 2026. The Clojure community's own defaults have moved to SSR + hypermedia (Biff's htmx stack, the rising Datastar SDK), so the book's thesis reads as the community's current direction rather than a contrarian bet — while every book competitor still centers the SPA or ignores the web entirely. • Production hardening is unmatched in the book market: strict compilation from commit one, a strict no-eval CSP, SRI + import maps + content hashing, Lighthouse gated in CI, and full CI/CD. No Clojure book — language or web — takes a single app all the way to an audited pipeline like this. • Coherence: one running application, every listing lifted from working code, prose and repo as one artifact. Competitors either survey many libraries (WdC), teach the language (Programming Clojure, Joy, Brave/True), or teach design in the abstract (Elements/Clojure Applied). None deliver a single, complete, deployable system. • Its dense, essayistic, 'options-then-justified-choice' voice is differentiated. Its nearest stylistic sibling, Elements of Clojure, is respected precisely for that register — and this book applies that register to an actual build, which nothing else does.

**Loses / disadvantages:** • Addressable market is narrow, and each opinion narrows it further. The audience is the intersection of: Clojure devs, who want SSR not SPA, who will adopt Datomic, who are past language basics, and who prefer owning code to using Biff/Kit. Datomic alone (commercial-heritage, small userbase even now that it is free) is a significant market constrictor that WdC deliberately avoids by using relational DBs. • It abandons the larger 'Clojure web' audience on purpose. A big share of Clojure-web interest is ClojureScript + re-frame; WdC 3rd ed serves that crowd and this book explicitly declines it. That is a defensible thesis but a real reach/sales ceiling. • No exercises, by design — a market disadvantage against Brave/True, WdC, and Programming Clojure, all of which give learners practice scaffolding. Density is identity here, but it also raises the on-ramp and shrinks the buyer pool. • No established brand or publisher moat. It competes against Pragmatic (WdC, Programming Clojure 4th, Clojure Applied), Manning (Joy), and No Starch (Brave/True) — plus core-team authorship on the two most current titles (Programming Clojure 4th ed, 2026). Discoverability, editing, and trust are steep hills for an independent title. • It cannot be anyone's first Clojure book — it assumes Brave/True-level fluency and comfort with Docker/TLS/reverse proxies. That caps it as a 'second book,' a smaller slice than the beginner on-ramp titles own. • Its strongest real-world competitor is free. Biff already ships the SSR+htmx architecture the book advocates, with less effort and active maintenance. The 'build it yourself so you own it' argument is honest but will lose the reader who simply wants to ship, and that is a large fraction of the pragmatic audience. • Timeless-reference risk: it is pinned to specific, fast-moving tooling (reitit, http-kit, Tailwind, ClojureStorm, Forgejo Actions, a vendored morph lib). The language books (Programming Clojure, Joy, Elements) age far more slowly, so this book's currency advantage today becomes a staleness liability faster.

**Positioning:** There is a real, verifiable gap, and this book sits squarely in it: as of mid-2026 no published book builds a current, from-scratch, no-framework, server-rendered, Datomic-backed, production-hardened Clojure web application — the nearest book competitor (Web Development with Clojure, 3rd ed, 2021) is SPA/re-frame-first, framework-assisted, relational, and five years stale, while every other strong title (Programming Clojure 4th ed 2026, Joy, Elements, Clojure Applied, Brave/True) teaches the language or design and never ships a web app. The book is also on-trend: it aligns with the community's own 2024-2026 move to SSR + hypermedia (Biff/htmx, Datastar), so its thesis is current-mainstream rather than contrarian. The catch is audience size. Every strength is also a narrowing choice — SSR-not-SPA, Datomic, no framework, no exercises, second-book difficulty — so the true addressable market is a premium niche: advanced Clojurists who have decided against the SPA, will run Datomic, and want to own their stack rather than adopt Biff. That is a real, monetizable audience for an independent/premium title, but it is small and it is not growing the way the ClojureScript/SPA readership is, and its most dangerous competitor (Biff) is free and embodies the same architecture. Correct positioning: sell it as the definitive advanced 'ship a real Clojure SaaS' book and the canonical written statement of the own-your-SSR-stack philosophy — explicitly the 'after Programming Clojure or Brave/True' book — not as a general Clojure web tutorial competing head-on with WdC for the mass market. In that lane it is best-in-class with essentially no current rival; outside it, the niche's own size is the ceiling.

**Wins:** • Only title in the lens that builds a COMPLETE production SaaS from empty directory to CI/CD, not a toy (Contact.app CRUD) or a framework feature-tour. No competing book defends its hypermedia choice at that resolution. • Genuinely novel domain angle: a history-shaped domain (recipe versioning — 'Git for recipes') whose versions, diffs, forks, and as-of reads fall out of Datomic's d/history and d/as-of rather than a hand-rolled audit table. Ch.2/Ch.14 show Biff and Kit already scaffold SSR+htmx and Datastar makes the same server-authoritative bet — so the book isn't contrarian in its own ecosystem; it's the rigorous, production-grade expression of that ecosystem's consensus, with a clean swap path (the server contract survives adopting htmx/Datastar instead).

**Loses / disadvantages:** • Community, ecosystem, and hiring pool are a fraction of every competitor's. htmx passed ~47k GitHub stars and is 'most admired' in State of JS, yet still only ~7% developer usage vs React's ~40% and a ~560:1 npm ratio; Rails and Phoenix each dwarf the Clojure community, which is itself small, and Datomic smaller still. A hand-rolled Clojure+Datomic stack is a niche within a niche — the book concedes the 'less common skill set than React' is a real hiring and onboarding cost. • On-ramp and addressable audience. Hypermedia Systems is free and uses Flask/Python that almost anyone can read; Rails 8 and LiveView open with generators and gentle walkthroughs. This book demands fluent Clojure, comfort with Docker/TLS/reverse proxies as prerequisites, willingness to think in Datalog, and appetite for a dense essayistic 'best-engineering' style. The steepest on-ramp and the smallest reachable readership on the shelf. • Batteries-included time-to-value. Rails 8 (Kamal, generators, built-in auth) and LiveView get a team to a live, interactive, deployable app dramatically faster than building a dispatcher, reload matrix, and inspector by hand. For teams whose goal is to ship, the frameworks win on velocity — and this book's 'build it yourself' is, on its own terms, more work. • Maintained vocabulary vs an owned seam. htmx, Turbo, and LiveView are maintained by teams, with upgrade paths, plugins, and a deep well of StackOverflow/tutorial answers. The book's hand-built dispatcher/registry/reload is yours to maintain forever; it frames that as ownership, but at team scale and across staff turnover it is a genuine bus-factor and maintenance liability the framework books don't impose. • Interactivity ceiling. LiveView (and React via Rappin's book) reach rich real-time and highly-interactive UIs out of the box; this book explicitly forgoes that (a round trip per interaction; real-time and offline reachable only as islanded features with their own sync). For any product that drifts app-shaped, the framework competitors offer a smoother path and this book tells you to switch models. • No mobile and no JSON-API story. Hypermedia Systems extends hypermedia to native mobile via Hyperview; this book has no mobile chapter. And it is pure page-granular SSR that grows only two tiny non-HTML endpoints (Ch.14) — teams needing a public or mobile JSON API for multiple consumers get essentially no guidance, the same gap a reviewer flagged in Hypermedia Systems, which at least devotes a chapter to it. • Not free, and single-stack. The free, language-agnostic-ish Hypermedia Systems and the massive Rails/Phoenix documentation corpora set a high bar for perceived value; a paid book locked to Clojure+Datomic+one hand-rolled architecture asks the reader to buy into three niche bets at once before the content pays off.

**Positioning:** There is a real, uncontested gap — but it is a deep-and-narrow one, not a broad-market one. On the hypermedia/SSR/anti-SPA shelf every existing title is either (a) a concepts-and-patterns book on a deliberately toy app (Hypermedia Systems), or (b) a batteries-included framework book that hands you a pre-made answer (Rails 8, Programming Phoenix LiveView, Modern Front-End Development for Rails). NONE builds a complete, production-hardened, security-first SaaS from an empty directory to CI/CD while showing the alternatives behind every decision, and none is in Clojure/Datomic mining immutable history as the product itself. That combination is genuinely unoccupied and defensible. The catch is audience size: the buyers are the intersection of 'reads Clojure fluently,' 'willing to run Datomic,' 'already accepts server-authority,' and 'wants a dense essayistic best-engineering treatment rather than a quick start' — a thin slice of an already-small language community (senior Clojure engineers, architects, and the Biff/Kit/Datastar crowd who've pre-made the thesis). This will never sell like a Rails book, and it shouldn't be positioned to try. Its strongest, most honest hook is that it is the ONLY rigorous, production-grade rendering of the Clojure community's own default stack (Biff/Kit/Datastar all scaffold SSR+htmx) — the definitive deep-end reference for people already on this road, not an on-ramp for people deciding whether to take it. The same density and hand-rolled machinery that are its identity are also the ceiling on its reach: accept that trade deliberately. Market it as 'the definitive Clojure server-authority SaaS build,' adjacent-shelf to Hypermedia Systems (which sells the philosophy) rather than competing with it for the newcomer — a reader could well buy Hypermedia Systems for the why and this book for a fully-worked, hardened how in their language.

### Datomic learning resources
**Wins:** • The java.time <-> java.util.Date bridge (ch.08): a complete, tested, reflection-clean solution (convert-instants/convert-dates, transact*/pull*/q*/as-instant) to the fact that :db.type/instant returns java.util.Date. Every real modern Datomic app hits this, and NO official doc, tutorial, course, or blog in the landscape covers it. This is close to a unique contribution. • basis-t vs :db/txInstant for version ordering (ch.09 version-history): sorting versions by (d/tx->t tx) rather than txInstant because two edits in the same millisecond tie on the instant and make 'latest' non-deterministic. A subtle, genuinely production-grade correctness insight surfaced almost nowhere else. • The upsert-through-identity foot-gun (ch.08): transacting a new entity carrying an existing :db.unique/identity value upserts and can silently rewrite a *different* identity attribute (a fresh :user/id onto the existing entity). Named, explained, and connected to why the auth path checks first — non-obvious and well taught. • History wired into a shipped feature: the only resource that takes d/history + d/as-of and turns them into version-history, version-as-of, line-diffs, and lineage/forks, with a URL-addressable, year-cacheable basis-t as the immutable version address, plus tests-as-spec against an in-memory fixture. Everyone else demonstrates the API in a REPL; the book demonstrates the product. • Production liabilities priced honestly: excision + GDPR right-to-erasure (history is not touched by retraction), :db/noHistory, unbounded history storage growth, the transactor topology dev never rehearses, and Pro-vs-Cloud trade-offs. Tutorials never bill these. • The counterfactual positioning: an itemized, both-directions migration cost vs PostgreSQL (versions table + triggers, temporal tables, audit-can-disagree-with-data) and a fair nod to XTDB v2 as the bitemporal alternative. No learning resource offers this decision framing. • Teaching prose that derives the model from SQL intuition — datoms as five-slot facts, joins as shared logic variables, database-as-immutable-value — more accessible than day-of-datomic's terse code and more grounded than the official overview. • Testability as a first-class Datomic pattern: the System/nanoTime-named datomic:mem fixture with get-connection redef gives a fresh DB per test in milliseconds — a pattern official material barely teaches and that most newcomers get wrong.

**Loses / disadvantages:** • It is not a Datalog tutorial and never becomes one. Ch.08 shows ~5 query shapes; a reader needing query fluency (rules, or/not, get-else, aggregates, recursive pull, richer :in binding) must still go to Learn Datalog Today or day-of-datomic. The book builds the EAV *mental model* but not query *skill*, and the interactive resources are strictly better for that. • No transaction-metadata / attribution. version-history captures :db/txInstant but never records who made an edit or why. day-of-datomic's provenance.clj teaches reified-tx attribution ('who changed the title and when') — the natural next question for an audit/versioning product, and one the canonical resource answers and the book does not. • Peer/Pro only. Jacek Schae's course covers Datomic Cloud end-to-end (Client API, dev-local, Ions, AWS deploy). The book gives connection/query *shapes* that carry over but explicitly punts the entire Cloud deployment lifecycle; a Cloud-targeting reader gets no operational guidance. • It under-engages the well-known community skepticism about using d/as-of for *domain* versioning (vvvvalvalval: 'model revision entities explicitly'). It asserts immutability-as-the-model confidently without weighing that named counter-position, and never flags the real schema-evolution-under-as-of caveat (reconstructing an old state with a pull pattern that references later-added attributes). A Datomic veteran would want that argued, not assumed. • No interactive practice or exercises (by design). For actually building Datomic/Datalog skill, the in-browser sandbox of Learn Datalog Today and the runnable REPL files of day-of-datomic beat dense prose plus a companion repo. • Narrow schema-feature breadth. It covers unique/identity, cardinality, ref, fulltext, and value types, but not composite/tuple attributes, :db/isComponent, transaction functions, or schema alteration/migration strategy at any depth. Official reference and day-of-datomic go wider for a reader who needs those. • Currency and authority: pinned to a specific artifact version and frozen at publication, where the official docs and example series track the live product (and are free).

**Positioning:** There is a real, specific gap and the book fills it — but a narrow one, and it is not the gap a Datomic *newcomer* has. The existing landscape bifurcates cleanly: (a) introductory/reference material that is free and authoritative but stops at 'the basics' (official Peer/Client tutorials, the unfinished 'Datomic by Example' TODO series, Learn Datalog Today for query syntax, day-of-datomic for canonical REPL patterns), and (b) exactly one substantial paid course (Jacek Schae) that is Cloud/Client-API and REST-oriented. NONE of them shows how to build a real, production-hardened, history-shaped application FEATURE on Datomic's temporal model, in prose, with tests, with the java.time bridge solved, and with the production liabilities (excision/GDPR, noHistory, storage, transactor) priced. That 'intermediate, applied, Peer/Pro, SSR' tier is essentially uncontested, and the java.time bridge and basis-t-ordering material are genuine contributions found nowhere else. So the book meaningfully advances Datomic pedagogy — but as a *complement* to Learn Datalog Today + the official tutorials, not a substitute: it presumes you can already think in EAV and does not build query fluency, so it cannot be someone's first Datomic resource. Audience size is small and stacked: Datomic is a niche within Clojure's niche, and this is 3 of 26 chapters, so it is not a 'Datomic book' and won't be bought as one. The honest verdict: the Datomic chapters are a strong differentiator and a real, underserved contribution for the specific reader building a history-shaped Pro/Peer SaaS — but they are a differentiator FOR the SaaS book, not a standalone market. The book's market case rests on the whole spine (SSR + Clojure + Datomic + hardening), and Datomic is one of its sharpest, least-duplicated edges rather than its foundation.

### The "build a real SaaS from an empty directory" genre
**Wins:** • Production hardening as a first-commit contract, not an afterthought chapter. No genre competitor treats security/performance this way: a strict, no-eval Content-Security-Policy that arrives strict and is never relaxed, plus SRI, an import map, content hashing, and Lighthouse scores gated at 100 (accessibility/best-practices) with a 95 perf floor in CI. Django for Professionals and Agile Web Dev have a security chapter and a deploy chapter; this book makes hardening a through-line and a CI gate. • Depth and honesty of decision-making. The genre's dominant pole is happy-path 'wire up what the framework gives you'; this book's every chapter is problem → naive option → rival engineer's option → justified choice with named costs. In spirit it out-reasons the tutorials and rivals Two Scoops, but as a single continuous build rather than a rules catalog — more reusable because you see each decision earned in context. • Bespoke developer infrastructure that no competitor ships. Competing books lean on the framework's dev tools; this one builds tooling as owned infrastructure, early. • A genuinely differentiated data story. Version history, point-in-time reads, forks, and diffs fall out of Datomic's `d/history` and `d/as-of` rather than a hand-rolled audit table. No SQL-backed Rails/Django/Phoenix competitor can match history-as-a-first-class-domain; the domain (Git-for-recipes) was chosen to exercise exactly this, and it's a real thesis no rival attempts. • The anti-framework, server-authority build is executed end to end by hand — SSR + progressive enhancement (every feature works without JS), a morph dispatcher for in-place navigation without a client framework, HMAC-signed single-use magic-link passwordless auth built from scratch rather than a plugin. Every competitor (Rails, Django, Phoenix, even Luminus) hands these to you; this book is the only one that makes 'own your code all the way down' the actual deliverable. • Uncontested within its own language at this depth. The only comparable Clojure book (Sotnikov) is a wider-but-shallower primer that teaches the opposite (re-frame SPA) architecture; Biff/Kit/Datastar are frameworks and docs, not a narrative production build. For a Clojure engineer who wants a hardened, server-authoritative, from-scratch SaaS, there is effectively no alternative book. • Proof, not just claims: Playwright e2e driving real registration/login/logout flows in a browser, plus a self-hosted CI/CD pipeline (Forgejo Actions, Podman, automated deploy) that builds, tests, audits (Lighthouse), and ships — an own-your-pipeline story stronger than most genre entries, which stop at a managed-PaaS deploy chapter.

**Loses / disadvantages:** • It does not deliver what the word 'SaaS' promises the genre's readers. The market's own definition (Bullet Train, Jumpstart, FullstackPhoenix, Rob Race) centers a SaaS on billing/subscriptions + multi-tenant teams. This book omits billing entirely (no payment provider, subscriptions, dunning, or proration) and reduces multi-tenancy to 'the tenant is the user' with no orgs/teams. A reader who buys it to learn how to charge money and onboard teams is under-served versus every SaaS-branded competitor. • No background jobs, scheduled work, or real outbound email. Race (ActiveJob/Solid Queue), Agile Web Dev (ActiveJob + Action Mailbox inbound), and the Phoenix kits all treat async work and transactional email as table stakes; here they're explicitly left as 'the obvious next infrastructure,' and dev email stops at Mailpit. • No observability, scaling, or backup/restore story. Metrics, distributed tracing, and alerting are gestured at, not built; the app is a single process against a single transactor with an in-process rate limiter honestly flagged as a seam; durability bottoms out at 'back up the Postgres behind the transactor yourself.' The genre increasingly expects at least a nod to running the thing in production over time. • The difficulty floor and prerequisites shrink the audience hard. It assumes fluency in Clojure plus comfort with Docker/TLS/reverse proxies as owned tools, and it front-loads a dense infrastructure chapter (three cert trust stores, container capabilities) before the app appears. Django for Professionals and Agile Web Dev welcome relative beginners; this book self-selects a narrow, senior slice. • Community, hiring pool, and ecosystem size are a structural disadvantage the book itself concedes. Clojure is a small pond; Rails/Django/Phoenix each carry orders of magnitude more readers, jobs, Stack Overflow answers, and library maturity. A Rails or Django SaaS book reaches a mass market; this reaches a niche of a niche (Clojure devs who also reject frameworks and starter kits). • By choosing 'own everything by hand,' it forgoes the very leverage the genre sells. A reader optimizing for time-to-launch is objectively better served by a starter kit (Bullet Train/Jumpstart) or Race's book: they'll have Stripe subscriptions, teams, and jobs running in a weekend. This book's payoff (comprehension, no black boxes) is real but back-loaded and only valuable to readers who weight ownership over shipping speed. • The bespoke tooling and hand-built plumbing (morph dispatcher, controller registry, construction view) are maintenance and comprehension burden that a framework would carry for you — a cost the book acknowledges but that a framework-convenience reader will read as reinventing wheels.

**Positioning:** There is a real and defensible gap, but it is narrow and the title oversells one axis while under-selling its true one. What this book genuinely occupies — and occupies almost alone across every ecosystem — is the 'from first principles, own every layer, hardened from the first commit, server-authority, no framework' corner. On that axis it is delivered unusually WELL: deeper per-decision than any tutorial, more hardened (CSP/SRI/import-maps/Lighthouse-in-CI) than any competitor including the pricey Rails/Phoenix kits, and carrying bespoke DX infrastructure and a Datomic history-as-domain story that no rival attempts. Within Clojure it is effectively uncontested at this depth; the only same-language book is a shallower, opposite-architecture primer. On the axis its NAME invokes — 'SaaS' — it is delivered unusually POORLY relative to the genre: the market defines a SaaS by billing, subscriptions, and multi-tenant teams, and this book concedes all three (honestly, in the afterword, but concedes them). It is really 'a hardened, server-rendered, framework-free production web app, built from scratch and understood all the way down,' not 'a SaaS business in a box.' Audience: senior, Clojure-literate engineers who value artisanal control and comprehension over shipping speed and who are actively skeptical of frameworks/starter kits. That is a small pond twice filtered — Clojure, minus the framework crowd — so mass-market size is the central risk. But the flip side is that no competing book serves that reader at all: the Rails/Django/Phoenix SaaS books can't (wrong language, framework-dependent, shallow on hardening), and the one Clojure book teaches the opposite thing. So within its niche the book dominates by default; across the broader genre it is a specialist, high-end entry, not a volume seller. The honest positioning fix is to lead with 'production-grade, server-authoritative, framework-free, hardened' and treat 'SaaS' as the vehicle rather than the promise — otherwise it will draw indie-hacker/ship-fast buyers who want Stripe and teams and will feel the omissions, while the readers it's actually perfect for (own-your-stack Clojure engineers) may not recognize themselves in a crowded 'build a SaaS' shelf.

### Voice / craft vs the best technical writing
**Wins:** • The essayistic problem/options/choice register is a genuine differentiator, not an affectation — because it is DDIA's own acclaimed structure (the single most-praised thing about the best systems book of the decade), because it is executed with real sentence-craft, and because it is load-bearing for the thesis: 'a decision you can see the alternatives behind is a decision you can re-make.' The form IS the argument. Very few technical books can say that. • Peak argumentative craft reaches genuine top-tier altitude. Ch.2 (positioning) is a masterclass: it synthesizes a decade of frontend history (React → hydration tax → islands → RSC → resumability → signals) into one through-line — 'a decade of extraordinary engineering has gone into paying that bet down for everyone else who inherited it by default' — pays honest, specific respect to every rival (Svelte, Qwik, LiveView, Biff, Datastar, Electric), and cites primary sources. This is DDIA-caliber synthesis and would survive in that company. • Intellectual honesty is a craft strength, deployed relentlessly and in both directions. Every choice names its cost ('the price we accept is that domain state lives in the database... a few interactions an SPA would keep purely local become a server round trip'), every technique names its limit ('one honest limit remains: the scan is still scoped by a hand-maintained namespace allowlist'). This is the exact virtue pathsensitive faults APOSD for lacking — the book fixes that defect structurally. • The register genuinely SUSTAINS; it never collapses into tutorial-speak. The test was the boring chapter: Ch.12 (Tailwind) still opens as an argument — 'where does the CSS live, and how do you keep it from rotting?' — and weighs BEM vs CSS-in-JS vs Bootstrap with real trade-offs before choosing. Even the mundane material carries the thesis. Quality does not decay from Ch.2 to Ch.20; the failure mode over 118k words is fatigue, not collapse. • Superb pedagogical framing of hard material. Ch.16 states the payoff before the machinery ('picture the bug these two chapters exist to kill... keep that one gesture in mind as the thing all of it is for') and acknowledges reader difficulty at exactly the right moments — the same technique reviewers praise as Nystrom's 'asides that acknowledge the reader's likely confusion.' The `> Decision —` / `> Lesson —` callouts give the dense chapters navigable spine. • Handles the book==code artifact's hardest problem — prose/code drift — more honestly than Crafting Interpreters. It explicitly declares the division of labor ('the prose carries the load-bearing reasoning... the repository carries the complete, current code') and promises to flag drift, rather than pretending the in-text snapshot is eternal. • Sentence-level craftsmanship is high and consistent: controlled parallelism and antithesis ('a choice you can see the reasons behind is one you can re-make... a choice handed down as an answer is one you can only obey'), confident periodic sentences, precise diction. The prose was clearly labored over, not dictated.

**Loses / disadvantages:** • No warmth, no humor, no personality — the single biggest craft gap versus the lovable end of the field. Nystrom's 'genuine wit,' his footnote jokes, his hand-drawn diagrams that 'signal a personal, crafted work,' and Evans' 'human talking to another human' voice are all absent by disposition. The book makes you respect it; it never makes you smile. Over 26 chapters that is a real cost, because affection is what carries a reader through a long book, and this book asks for admiration instead. • Relentless, unventilated density with no rhythmic relief. DDIA sustains its length by breaking the argument with concrete named-company war stories and a more generous cadence; this book has almost no such release valves. Every paragraph is balanced, every clause load-bearing, every sentence at maximum information density. The reader never gets a throwaway line, a breather, or a laugh. The fatigue risk over 118k words is not that the writing gets worse — it doesn't — but that the sameness of the high-effort register wears the reader down. • The prose leans heavily — arguably too heavily — on two devices: the em-dash parenthetical (multiple per paragraph, page after page) and the 'not X; Y' antithesis. Individually elegant, in aggregate they become a detectable tic. A craft editor would flag the monotony: the machine-even cadence, where every sentence lands with the same balanced weight, starts to feel airless precisely because it is so uniformly polished. The very consistency that makes it impressive also drains it of the human unevenness that makes Nystrom and Evans feel alive. • Long periodic sentences actively resist skimming and punish the tired reader. Some run 60+ words with stacked em-dash asides and semicolons (the Ch.1 difficulty paragraph is one enormous block). Kleppmann is praised as 'lucid' and skimmable; this is the opposite by design. The book even concedes the point — its hardest chapters are 'built to be read twice' — which is honest but is also an admission that first-pass comprehension isn't guaranteed. Ousterhout's approachable brevity and Pragmatic's punchy tips both beat it badly on 'absorb the point fast.' • The 'rival engineer' and the 'naive option' are argued abstractly rather than dramatized. Pragmatic makes ideas stick with named parables (broken windows, stone soup); DDIA makes them stick with named real systems. This book's options are weighed in the abstract, so it produces no detachable, portable, sticky artifact — no metaphor you repeat in a standup a year later. Its memorability is gestalt, not quotable, which is a weaker form of stickiness. • Visual craft is functional, not delightful. The austere monospace SVG diagrams and the dev-environment screenshots do their job but carry no personality and add no warmth — the exact opposite of the hand-drawn illustrations reviewers credit as central to both Crafting Interpreters and Wizard Zines. In a book that is otherwise fighting for the top tier on prose, the visuals are the one dimension that doesn't compete. • Occasional tip into the sententious. A few lines ('a choice handed down as an answer is one you can only obey') flirt with grandeur beyond what the local point needs. The book earns it most of the time through genuine substance, but the mannered register has a ceiling above which it reads as self-consciously literary for a technical text.

**Positioning:** On pure craft, the book ranks in the top decile of technical writing and is legitimately in the conversation with DDIA and Crafting Interpreters — a real achievement, not a flattering self-assessment. The market gap it fills is specific and genuine: there is no other book that applies DDIA's problem/options/justified-choice register, at DDIA's level of argumentative rigor and intellectual honesty, to a single end-to-end production build (the Crafting-Interpreters 'book IS the code' contract) in the Clojure/SSR niche. That triangulation — DDIA's essay-craft × Nystrom's book==code integrity × a working full-stack SaaS — is unoccupied. The register is a true differentiator, it sustains without collapsing, and it is the correct register for this book's thesis. But the craft ranking is uneven across dimensions and the audience follows from that: the book wins decisively on rigor, honesty, structure, and sentence-level control, and loses decisively on warmth, humor, ventilation, skimmability, and visual delight. It is writing to admire, not to curl up with. That caps the addressable audience at the self-selecting minority who actively want a dense, opinionated, essayistic engineering argument and will trade friendliness for completeness — a real and underserved audience (the same readers who finished DDIA and wanted one built all the way down), but a narrower one than a warmer or more skimmable book of the same rigor could reach. The density is IDENTITY, correctly, and should not be sanded down; the honest craft risk to watch is fatigue, and the cheapest mitigations that would not betray the identity are (a) modulating the em-dash/antithesis tic for rhythmic variety and (b) letting a rare human aside or concrete war story break the airless evenness — the two things DDIA does that this book, at its best, still does not.

### Audience size, commercial viability, and market risk
**Wins:** • Genuinely uncontested gap: there is no other book that combines Datomic + server-side rendering + no-client-framework + production hardening in one built application. The nearest title (Web Development with Clojure 3e, 2021) teaches the opposite stack (re-frame SPA, SQL) and predates the htmx wave; the only other Datomic-in-print reference (Professional Clojure) is a 2016 survey. The shelf is effectively empty. • Rides two real, independently-verified tailwinds at once. (1) The hypermedia/SSR resurgence is not hype-only: State of JS 2025 (pub. Feb 2026) shows htmx holding 'most admired' while React satisfaction hit its lowest despite 83.6% usage; htmx sits ~47k+ GitHub stars; 'JavaScript fatigue' is a documented, sustained driver. (2) Datomic became free of licensing fees in 2023 (Pro Apr, Cloud Jun; Apache-2.0 binaries on Maven Central) — the historical $5k/yr + AWS paywall that kept people away is gone, so the moment to write *the* Datomic book is arguably the best it has ever been. • Sells with the grain of the Clojure community's own defaults, not against them. Biff ships htmx-first, Kit has an official htmx module, and Datastar (SSE+signals, official Clojure SDK) is the emerging second pole. The 'server is the authority' position is the community mainstream, so the book is not a contrarian bet within its own ecosystem — it's the deep, own-your-code version of what Clojurists already reach for. • The audience, though small, is unusually high-value and high-intent. Stack Overflow 2024 makes Clojure the highest-paid language (median ~$95.5k) with a deeply experienced base (~12 yrs); State of Clojure shows 73% use it at work and 70% are 'very likely' to recommend with only 8% negative. This is a work-funded, senior, depth-appreciating readership with high willingness to pay a premium for rigor — the ideal buyer for a dense, expensive niche book. • Format is defensible against both AI and free blog content. 'One real app, every listing lifted from running code, the companion repo *is* the application, every screenshot is the live server' plus essayistic problem→options→justified-choice reasoning is exactly the architectural-judgement layer LLMs convey least convincingly and that scattered tutorials never assemble into a coherent whole. The value is the accumulated decisions, not the code — hard to substitute. • The book has already priced its own biggest risk into the text. Chapter 8 uses free Datomic Pro with datomic:mem in dev (zero infrastructure, no AWS, no cost to the reader) and itemizes the Postgres-plus-triggers and XTDB-v2 escape hatches honestly, preserving a 'tenancy waist' that survives the swap. That defuses the most common objection at the point of sale rather than pretending it away.

**Loses / disadvantages:** • The addressable market is a compounding intersection, and each ∩ multiplies the pool down. Clojure is genuinely niche (not in TIOBE's top 50; ~#46 IEEE Spectrum; ~0.01% language market share, ~1,500 companies tracked). Datomic is a minority choice *within* that niche (many Clojure web devs default to Postgres/next.jdbc or increasingly XTDB). SSR/no-framework is a live but self-selecting subset. And 'willing to read 118k dense words with no exercises' filters again. Realistic serious-reader ceiling is low thousands lifetime — plausibly hundreds to ~1–2k — which self-publish economics (Leanpub ~80% royalty; a cited comparable earned ~$17k on ~1,050 copies) translate to a modest sum, not a primary income. • Its single most distinctive feature is welded to its single largest risk. The history-shaped payoff — versions, diffs, forks, and point-in-time reads falling out of d/history and d/as-of 'for free' — only lands fully on Datomic. A reader who balks at Datomic (proprietary/source-closed, still 1.0-series with only incremental releases since 2020, Nubank-controlled, and carrying the well-worn 'won't build core infra on a closed DB that could get bought-out-and-killed' fear) loses the very thing that differentiates the book and must mentally port chapters 8–9. Documented migration paths (e.g. Multiply: Datomic→XTDB→Rama) show this churn is real. • It is disadvantaged on distribution and credibility versus the incumbent. Web Development with Clojure has Pragmatic Bookshelf behind it — catalog placement, review channels, search authority, and the default-recommendation slot in the community. A presumed self-published title starts with none of that reach and must win readers one word-of-mouth at a time, which caps discovery independent of quality. • Deliberate density with no exercises maximizes respect-per-reader but minimizes conversion breadth. The book's own primer flags spike chapters (devcontainer ops, java.nio file-watching, Datalog, the three-chapter ClojureStorm construction view) and tells readers to 'read the hard chapters twice.' That is identity, not defect — but commercially it means a high bounce rate among all but the most committed, shrinking the funnel further than a gentler book would. • A slice of the target readers will prefer to *buy* the plumbing the book insists on hand-building. Where the book builds its own morph dispatcher, inspector, and lifecycle registry ('own your code'), Biff/Kit/htmx/Datastar hand equivalent capability off the shelf for free. Readers who value time-to-ship over ownership are better served by those, and the momentum behind Datastar/htmx risks making 'build your own morph layer' read to some as reinventing a solved problem — a positioning headwind the book must keep actively arguing against. • The community is stable and mature but shows little sign of mainstream *growth*, so the pool is unlikely to expand under the book. Signals point to a seasoned, geographically concentrated (US/Brazil/Germany/UK ≈ 50% of respondents), Nubank-anchored base rather than a rising tide of newcomers — good for durability of a small audience, poor for any bet that depends on the market getting bigger over the book's shelf life.

**Positioning:** The gap is real and the bet is sound — but it is deliberately, structurally small, and should be judged as a niche flagship, not a volume play. There is a true uncontested position: no competing book pairs Datomic with server-rendered, framework-free, production-hardened Clojure, the one nearby title teaches the opposite (SPA/re-frame) and is dated, and the book rides two independently-verifiable tailwinds — the htmx/SSR resurgence (htmx 'most admired' in State of JS 2025 while React satisfaction fell) and Datomic going free in 2023 (removing the paywall that historically kept readers away). It also sells with the grain of the Clojure community's own SSR defaults (Biff/Kit/Datastar) to a high-value, highest-paid, work-funded, depth-loving audience. That combination is why the book can exist and be respected. The ceiling, though, is set by compounding intersections: Clojure is niche (outside TIOBE top 50, ~0.01% share), Datomic is a minority within it, no-framework SSR is self-selecting, and the dense/no-exercises format self-selects again — realistically a low-thousands (more likely hundreds-to-~2k) lifetime serious readership, which under self-publish economics is a meaningful-but-modest sum and a strong credibility/authority/consulting-lead asset rather than a primary income. The load-bearing commercial risk is the Datomic coupling — it is simultaneously the book's chief differentiator and its chief deterrent — and the author has handled it about as well as it can be handled (free Datomic Pro + datomic:mem, an itemized Postgres/XTDB escape hatch). Net: a well-judged, durable, premium niche book for a small expert audience; expect authority and respect, not scale. The right expectation is a canonical reference for the few hundred-to-few-thousand people it is precisely for, not a broad seller.

## 7. Appendix — full per-cluster close-read

### The Wager (preface, 01-primer, 02-positioning) — 8.5/10
*A genuinely strong, well-researched thesis statement that mostly earns its density and its confidence. Chapter 2 is the intellectual core and largely succeeds: the history is accurate, the traditions are disaggregated with real care, and the opposing case is steelmanned rather than knocked down. The soft spots are all load-bearing rather than cosmetic: the "best build" principle treats a four-way objective as a scalar, the "everything is converging on server authority" narrative is selectively told, and the headline "addressability" win meets modern meta-frameworks at exactly the point where they are strongest. None are fatal; all are the kind of thing a serious review would dwell on.*

**Strengths:**
- Ch.1 domain justification is the best thing in the cluster: the recipe app is defended not by novelty but by two structural properties -- it is 'history-shaped' (versions/forks/diffs ARE the product, so they fall out of d/history and d/as-of rather than a hand-rolled audit table) and 'content-heavy and read-mostly' (exactly SSR's best workload). This makes the demo an honest test of the stack rather than an arbitrary vehicle, and it is rare for a book to argue its example this rigorously.
- Ch.2 'Why some roads never made the bet at all' refuses the lazy move of lumping Vue/Svelte/hypermedia in with 'SPA frameworks that came back.' The Svelte-was-newsroom-graphics correction ('conflating the two is the specific error worth correcting') is non-obvious, correct, and argued against the author's own rhetorical convenience. This is where the chapter earns the reader's trust.
- Ch.2's steelman of the opposing case is clean and honest: three named categories where a SPA is genuinely correct (you don't own the backend, the product is app-shaped, offline-first), conceded without hedging, followed by an unflattering cost list for its own choice (a round trip per interaction, discipline that is 'cultural rather than enforced by a compiler,' a less common skill set = real hiring cost).
- Ch.2's Clojure-community grounding (Biff and Kit ship/reach server-HTML+htmx; Datastar as the rising server-authoritative pole; Electric as the admired radical bet) pre-empts the 'contrarian' charge by showing the position is the community's own default, and closes with genuine humility -- 'we choose the simplest expression of server authority, not the only one.'
- Ch.1's difficulty section is unusually honest reader-service: it names the four specific spikes (infra-dense environment chapter, java.nio file-watching, Datalog's EAV shape, the construction-view cluster) and tells the reader concretely how to handle them -- lean on the running repo, read the hard chapters twice for argument-then-mechanics. Most books hide their hard parts; this one maps them.
- The prose craft across all three chapters is consistently high -- controlled, essayistic, and load-bearing rather than decorative; the argument is carried by sentences, not by section headers.

**Weaknesses:**
- Ch.1's load-bearing principle treats 'best' as a scalar while defining it as four criteria -- 'the most correct, the most secure, the fastest, and the most pleasant to maintain' -- that routinely conflict (fastest vs. most maintainable; most secure vs. most pleasant early, e.g. strict-CSP-from-day-one). It declares this 'the only tiebreaker' but never says what breaks ties AMONG the four. The principle is rhetorically clean and philosophically under-specified, and 'the best... we know how to build' quietly deflates the objective framing to 'our considered opinion' without acknowledging the retreat.
- Ch.2's spine -- 'a decade of extraordinary engineering has gone into paying that bet down' -- is selectively told. Signals/fine-grained reactivity (Solid, Svelte runes) and compile-away-the-vDOM are largely about client-side update cost and DX for genuinely interactive apps, not about 'recovering server authority'; folding them into the penance narrative overreaches (Solid still hydrates; signals are about update granularity, largely orthogonal to the hydration bill the section files them under). The chapter wants it both ways: the whole decade was undoing one mistake (lines 25/45/65), yet a large class of apps is correctly a SPA (line 53) -- in which case those innovations are advances on a valid road, not debt repayment.
- Ch.2 leads its 'why the server' case with 'addressability by construction' as the signature free win -- but by the book's own account of 'the war to undo it,' addressability is the MOST-recovered cost: file-based routing already makes the URL the state, loaders already put data-fetch on the server, and Remix/SvelteKit forms already 'work without JavaScript and are merely enhanced.' A modern meta-framework app has addressability essentially by construction too. The genuinely durable differences (no model duplication, no hydration tax, no second build artifact, less ecosystem lock-in) are stronger ground; by headlining addressability the chapter meets its most serious rival at that rival's strongest point without saying so.
- The preface-to-ch.1 seam is the weakest structural join: both open by defining against 'the tour/tutorial,' and the preface's prerequisites paragraph plus its difficulty preview substantially duplicate ch.1's 'What you should already know' and 'How hard this gets' -- roughly a quarter of a four-paragraph preface is a trailer for the chapter that immediately follows. The 'tutorial' foil is also a soft target used twice: a serious reader's real alternative is other production-grade patterns, not the toy tour, and the book flatters itself slightly by repeatedly measuring against the weaker genre.
- Ch.1's 'journey' section (lines 33-43) is largely a table-of-contents rendered as prose -- dense enumeration of nearly every feature ('content hashing, Subresource Integrity, an import map, and a strict Content-Security-Policy...'). The 'five movements' framing rescues part of it, but for a reader who has the SUMMARY it is the one stretch of the cluster that invites skimming.

**Underdeveloped / raised-not-paid-off:**
- Datastar is named as a rising server-authoritative library with an official Clojure SDK -- arguably the closest existing thing to what the book spends chapters hand-building -- and then passed by. A Clojure reader immediately asks 'so why not just adopt Datastar?' The htmx comparison is explicitly deferred to ch.14, but the Datastar-specific 'why build instead of adopt' is answered only by the general 'own the moving parts' line; the single most obvious rival in its own community gets a nod, not an engagement.
- 'Morphing, prefetching, and optimistic updates hide most of the latency' (ch.2, round-trip cost) -- optimistic updates reintroduce exactly the client-side prediction/state the book elsewhere presents server authority as sparing you. The tension is raised in one clause and never examined; a skeptical reader notices the escape hatch quietly smuggles back a little of what was disavowed.
- The multi-objective tie-break behind 'the best build' (from weakness 1) is the thing a discerning reader most wants resolved before accepting the principle that governs the whole book, and it is left implicit -- the reader has to trust that the four criteria never seriously collide, or that the author's taste silently arbitrates when they do.

**Voice & pacing:** Register is elegant, dense, and em-dash-driven essayistic, held consistently across all three chapters; it reads aloud well and the density mostly earns its keep -- the history in ch.2 IS the argument, not ornament. Two pacing notes. First, preface and primer are necessarily all-promise-before-payoff; that is fine for openers but front-loaded, and the near-duplication between them makes the run-up feel longer than four paragraphs plus a chapter should. Second, ch.2 piles superlatives thickly -- 'extraordinary engineering,' 'the most serious attack on the hydration tax anyone has mounted,' 'the purest statement of the era' -- so the reader is repeatedly asked to accept 'the most X' claims on the author's authority; most are defensible, but the accumulation is a stylistic risk for exactly the skeptical reader the book is trying to win. The one genuine dead spot is the ch.1 'journey' section, which reads as a TOC in prose.

That inconsistency of posture is un-interrogated and undefended.
- Ch.12's 'options' section is the weakest steelman in the cluster. CSS-in-JS is listed only to be dismissed in two sentences ('not applicable'), and the genuinely live rival for a zero-dependency Clojure book — a small hand-authored design-token CSS file (or Garden) with no npm, no scanner, and no dynamic-class footgun — is never named, even though the chapter's own hand-written-CSS section is a partial admission that 'utilities for everything' isn't the real position. This connects to a thesis-level tension the book leaves open: Ch.11 rolls its own i18n specifically to avoid a dependency and 'understand every line,' while Ch.12 adopts an external Node CLI with a config model you must learn and a silent failure mode — and the flip in the dependency calculus between adjacent chapters is never reconciled.
- Ch.12 identifies a silent, 'no compiler or test flags the gap' failure mode and then guards it with developer discipline alone — the exact 'by discipline, not by construction' posture Ch.10 opens by rejecting. Given the book has a whole testing chapter and a 'hardened' ethos, proposing zero mechanical guard (not even a grep-lint for `(str "bg-` fragment patterns) is a raised-and-not-paid-off, and it reads as a cross-chapter register inconsistency.

**Underdeveloped / raised-not-paid-off:**
- Ch.10 names the hardest testing question in the cluster — email sending and Datomic transactions are 'harder to unit test' and are the stated reason the coverage bar is only 50% — and then defers it entirely to later integration/e2e chapters. The one genuinely difficult unit-testing problem the chapter raises is the one it doesn't tackle on the page.
- Ch.10 leans on Cloverage line/form coverage as *the* coverage story and gestures at the metric's weakness ('don't write bad tests just to hit a number') without taking the next step: line coverage measures execution, not assertion strength, so a 50% gate can be green with weak tests. The critique is implied and dropped.
- Ch.10's two most prominent teaching listings — the `routes-test` namespace and the entire `myapp.inline-tests`/`parse-port` example — are, by the chapter's own admission, invented for the page and not in the repo. The book is transparent about it, but in the one chapter whose subject is 'test the real thing,' leaning on constructs that don't exist in the app it tests is a mild tension with the 'the repo IS the app' promise that a careful reader will feel.
- Ch.11's parity-test failure output is diagnosed as poor UX (it dumps both 80-key sets) and the fix offered is a manual REPL `clojure.set/difference` — but the chapter never just improves the test to print the symmetric difference itself. It spots a weak test and points at a workaround instead of showing where the test's own diagnostics can carry the load.
- Ch.11 notes detect-locale collapses regional variants (`.getLanguage` maps both `nl-BE` and `nl-NL` to `:nl`) as 'exactly right' for two languages, but doesn't connect that this is the seed of the very regional-variant problem the 'why not a library' section is about — a one-line link would tighten the arc.

**Voice & pacing:** Register holds strongly across all three chapters — the dense, essayistic, name-the-cost rhythm that is the book's identity is intact, and the closers of Ch.11 and Ch.12 are the high points, prose that earns its density with pure signal. Ch.12's token/prefix and dynamic-class passages are tight; Ch.11's fallback-chain and interop-note passages read cleanly. The unevenness is concentrated in Ch.10, which oscillates between straight infrastructure narration (the request-builder `cond->` walk, the line-by-line config-test reading — competent but closer to narration than argument) and a genuinely virtuoso middle (the in-file test macro). The two halves don't share a thesis, and the opening claim 'None of this is exotic. That is the point' actively undersells the chapter's best and densest material. No passage is unclear or over-explained; the issue is proportion and framing, not readability.
  
**Standout:** Ch.11's "Why not a library?" section — specifically the reframing that the primitives' fragilities (fixed argument order from `format`, fixed word order from concatenation, fixed field order from the DateTimeFormatter pattern) "stay invisible exactly as long as the shipped languages happen to agree," landing on: "The moment a translator who does not read Clojure needs to move an argument inside a sentence, format strings and concatenated fragments stop being primitives and start being bugs." Most books justify rolling their own with "it's simpler"; this one enumerates the three features it already uses, prices each, and names the precise condition under which its decision inverts. That is the whole book's method executed perfectly.

### Views+morph (13-hiccup-views, 14-morph-dispatcher) — 8/10
*A strong, dense, intellectually honest cluster that mostly lives up to the book's "options behind every choice" standard. Ch.13's escaping argument and Ch.14's htmx decision are near best-in-class: they steelman rivals, name the decisive reason, and concede what they cost. The prose earns its density and the correctness claims hold under source inspection (I checked several subtle ones). What keeps it from a 9 is a single genuine gap that runs against the book's own creed: the one dependency the whole cluster rests on — idiomorph — is treated as a black box. Its state-preservation is asserted, not shown, and its alternatives (morphdom/nanomorph) are never surveyed the way templates and htmx are. That, plus Ch.13's heavy forward-reference tax, are the real editorial dents.*

**Strengths:**
- Ch.14 'Decision — why not htmx?' is the standout of the cluster and genuinely steelmanned: it explicitly disavows a line-count dismissal, draws the real architectural line (element-granular hypermedia grows a fragment-shaped server API; this app's contract is page-granular), concedes that used at page granularity htmx 'collapses to hx-boost plus a morph extension,' names Biff/Kit/Datastar as 'good answers whose trade is the inverse of ours,' and admits they already vendor idiomorph — 'the one piece of that family we could not improve on.' That last concession is the mark of an honest argument.
- Ch.13's 'why HTML-as-data, not templates?' box refuses the easy dismissal of Selmer/Enlive: it credits their real strengths (a designer can edit them, MDN snippets paste in unchanged) and lands the decisive, non-generic reason — 'data can carry metadata; strings cannot' — which pays forward concretely to the inspector welding file:line:column onto element vectors and the construction view walking recorded Hiccup. The choice is earned, not asserted.
- Ch.13's output-encoding section is concrete where it counts: it shows the actual escaped bytes for BOTH contexts (element text `&lt;img...&gt;` and the attribute breakout `value="&quot; onmouseover=&quot;..."`), correctly positions CSP as defense-in-depth 'sitting behind escaping,' and disciplines every h/raw site to an auditable rule ('grep for h/raw, every hit is a literal you wrote or a renderer that sanitizes'). The one untrusted raw site (markdown) is called out by name.
- Ch.14's graceful-degradation-by-method reasoning is precise engineering prose: a GET network failure falls back to a real navigation so the user isn't stranded; a failed POST deliberately does nothing, and the chapter explains why replaying it would be 'wrong twice over' (a navigation is a GET and can't carry the body; re-issuing risks a duplicate write because the error can strike after the server already processed it). This matches the source exactly.
- Ch.14 is scrupulously honest about the limits of its own mechanisms: the in-flight dedupe 'claims no more than' per-destination (clicks to two different destinations still race), and it works through what the losing race order leaves behind — address bar and page stay mutually consistent, and one Back restores the newer click. I traced the pushState ordering against the source and the claim is actually correct, which is impressive for a throwaway sentence.
- The two chapters are welded together with real load-bearing seams, not just cross-links: `<main data-layout>` is planted in Ch.13 with a stated future purpose and cashed out in Ch.14's cross-layout guard ('This is the payoff for putting data-layout on every <main>'); the 'handlers do not branch on request type' discipline in Ch.13 is what makes Ch.14's 'the server keeps rendering in blissful ignorance' true. The architecture is coherent across the boundary.

**Weaknesses:**
- The morph engine itself is the one major choice in the cluster whose alternatives are never surveyed. Templates get Selmer/Enlive; the dispatcher gets htmx/Turbo/Unpoly/Biff/Kit/Datastar — but idiomorph is simply adopted ('we vendor it,' 'the htmx project's own morphing engine') with no word on why idiomorph over morphdom or nanomorph. I confirmed morphdom/nanomorph appear nowhere in the manuscript. For a book that promises the options behind every choice, the single most load-bearing dependency arrives as a given.
- Relatedly, Ch.14's central capability is asserted where it should be shown. 'Morphing diffs the existing DOM against the new DOM and applies the minimal set of mutations, so focus, scroll position, form state, and ongoing CSS transitions inside unchanged subtrees survive the update' — but the mechanism that makes this true (idiomorph's id-set matching, which is exactly what distinguishes it from a naive innerHTML swap that would blow all of that away) is never explained. A skeptical reader who has used morphdom is told the outcome and asked to take the how on faith.
- Ch.13 carries a heavy forward-reference tax. The asset-pipeline chapter (Ch.24) is invoked ~6 times for machinery that is load-bearing in THIS chapter's listings — asset URL resolution, SRI integrity, the import map, the CSP hash-matching that justifies every h/raw on inline scripts, and defn-asset's register-inline-script!. A reader working Ch.13 must take a lot of central plumbing on faith 11 chapters early. The book flags this ('drop the names you have not built yet'), but the reading-experience cost is real and concentrated here.
- Ch.14's 'A note on the accept header' section is slightly self-undermining. It correctly warns 'do not build server logic around it,' but by calling the header 'vestigial — harmless' it draws attention to code the prose deems pointless without resolving whether it should exist at all. The section's real value is forestalling a content-negotiation misconception; framing the header as dead-but-harmless muddies that with a small 'then why send it?' the chapter never quite closes.

**Underdeveloped / raised-not-paid-off:**
- Why idiomorph specifically — the id-based matching algorithm and how it beats morphdom/a naive diff — is the obvious question a working engineer brings, and it is never opened. The morphCallbacks hook and ignoreActiveValue are described as pass-throughs to idiomorph without ever characterizing what idiomorph does with them.
- Ch.13 lists a separate top-level <script> tag per behavior module (controllers.js, live-form.js, defer-details.js, ...) alongside the controller registry, but never justifies the granularity: why does each behavior mount its own <head> script rather than being imported by controllers.js? It's presented as a convention ('each later behavior chapter adds its module here') without the design reason a curious reader wants.
- The cross-layout guard only fires when both live and incoming <main> carry data-layout AND they differ; the code silently falls through to a normal morph if an incoming page lacks data-layout. The prose presents the happy path as complete and doesn't acknowledge that edge, small as it is.

**Voice & pacing:** Register is consistent and confident across both chapters, and the density is identity, not padding — nearly every paragraph carries an argument or a concrete mechanism. Ch.14 is the tighter, more argumentative of the two and reads beautifully ('Build the dispatcher once, correctly, and two later chapters get to treat it as bedrock'). Ch.13 is marginally more uneven: the escaping and decision sections are essayistic and earn their length, but the closing 'middleware stack' section reads more like reference material (a stack listing plus three bullets) and sits slightly apart from the chapter's argumentative spine. Both chapters lean hard on forward references — Ch.13 especially — which occasionally makes a section feel like a promissory note rather than a settled account, though the payoffs I checked (Ch.18, Ch.19) do land.
  
**Standout:** Ch.14's 'Decision — why not htmx?' box — specifically the element-granular-vs-page-granular framing, made honest by three concessions in a row: that at page granularity htmx 'collapses to hx-boost plus a morph extension,' that Biff/Kit/Datastar are 'good answers whose trade is the inverse of ours,' and that they already vendor idiomorph, 'the one piece of that family we could not improve on.' It is the cluster's clearest demonstration of the book's method: steelman the rival, name the real architectural axis, and admit exactly what you borrowed from it.

### Dev-tooling cluster (15-inspector, 16-construction-view, 17-overlay) — 8/10
*This is the most technically ambitious stretch of the book and, at its best, its most original: the "welded coordinate" (source location riding on the runtime Hiccup value via tools.reader + the compiler's metadata preservation) and its dual (execution-as-data via a recording compiler, keyed on the same string) are a genuinely novel, correctly-argued pair of ideas that no front-end framework offers. The writing peaks are outstanding — the recipe-title bug story, the 30GB-leak retreat, the expression-scan-vs-name-allowlist lesson, and flow mode's Whyline-frontier honesty. What keeps it from best-in-class is a cost/benefit argument that is asserted and then half-conceded rather than fully won: three chapters — the largest single feature in the book — build a bespoke in-page tool atop FlowStorm (which already has a debugger UI) and couple the headline payoff to VS Code + Joyride, and the chapters under-earn two things they spend heavily on (ch.15's reverse direction and ch.17's six dossier drill-downs). The density is identity, not a defect, but the run tests reader patience where the payoff is thinnest.*

**Strengths:**
- Ch.15's keystone is shown, not asserted: the two 'stacked facts' (tools.reader attaches :line to vector literals at every depth; the compiler preserves a vector literal's metadata onto the runtime value even for vectors built in a loop) are each demonstrated with a tight, load-bearing REPL snippet — the (map meta (render [1 2 3])) => ({:line 9}{:line 9}{:line 9}) example makes the whole feature's possibility concrete in three lines.
- Ch.15's CLJ-2568 discussion (why add-file-meta can be a plain postwalk on Clojure 1.12 but must be hand-rolled earlier, because postwalk silently dropped seq metadata until 1.12, and that metadata feeds compiled line numbers) is exactly the kind of deep, correct, non-obvious engineering detail that earns density.
- Ch.16's 'Why not just extend the inspector?' is a real steelman-then-defeat: var-wrapping yields an accurate tree of named-function edges but is structurally blind to sub-expressions like (:user/display-name (:recipe/user recipe)) — a concrete, honest reason the compiler-level recorder is needed rather than a strawman.
- Ch.16's 'instrument the boundary you can name, but read the expressions you can't' is the single most insightful idea in the cluster: a name-allowlist of db wrappers is blind to raw d/q / d/as-of / d/history (never compiled by us, so no :fn-call recorded) — exactly the time-travel reads a recipe-versioning app is built on — so it detects Datomic ops by expression shape and resolves each back to real Datalog-as-data. The lesson generalizes cleanly.
- The 30GB memory-leak narrative (record-from-boot was tried, shipped, and backed out because ClojureStorm timelines are append-only; recording is now gated to one render, with -Xmx2g as a loud backstop) is honest engineering history that justifies the per-page gating instead of merely asserting it.
- The synchronous-handler correctness argument (http-kit one-thread-per-request, domain called inline, hiccup2 forces eagerly, therefore a request == exactly the slice [start,end) of one thread's timeline) is load-bearing and honestly flagged as the thing to revisit the day an async handler appears.
- Ch.17's flow mode is a genuinely new capability and beautifully narrated as a single keyboard moment (Alt+click the third card; flow resolves that instance, walks its ancestor path, eid-matches the read that produced it, returns 'green and unhelpful,' pivots to d/history) — and its limit is stated with rare intellectual honesty (eid-matching is identity not lineage; true taint-tracking is the research frontier, with a Whyline citation and 'we stop one honest step short').
- N+1 detection lands a real, non-toy payoff: the honest false-positive (pull-per-card on the index is a Datomic peer local index walk, 'information, not an indictment') sits beside the genuine catch (find-user-by-email run twice per request by stacked auth gates, fixed by resolving the user once onto the request).
- Consistent production-absence discipline across all three chapters — tag-root macro vanishes, dev-only namespaces under dev/, requiring-resolve → 404 in prod, storm property gating — argued piece by piece rather than hand-waved, matching the book's 'structurally absent, not merely disabled' bar.

**Weaknesses:**
- The whole cluster's cost/benefit is asserted then half-conceded rather than won. Three chapters — the largest single feature in the book — build a ~900-line overlay + 8 endpoints + a compiler swap on top of FlowStorm, which already ships a full time-travel debugger UI. The novel justification (DOM correlation via the welded coordinate; click-the-pixel per-instance) is real, but ch.17's 'What this investment buys, and when' blockquote confronts the tension and then concedes the everyday case to 'a println and a REPL,' landing the payoff on the narrow 'day you hunt a wrong value across identical components.' For the book's biggest tool, that is an honest but thin return, and the reader is left doing the cost/benefit the chapters should close.
- Ch.15's reverse direction (code→element) is under-motivated relative to its machinery. Element→code has an obvious, constant use; code→element gets an index, a Joyride agent, WebSocket role-tagging, a DOM-as-truth highlighter, and echo-guards spilling into ch.17 — yet no concrete bug story answers 'when do I actually put my cursor on a view fn and need the browser to light up?' The intro even calls it 'rarer even there,' which reads as a tell that its value is less than its build cost.
- The editor coupling narrows the book's own 'SSR gives up nothing' thesis more than the chapters admit. The headline payoffs (click-to-open, cursor-to-browser) require VS Code + Joyride + Calva; the whole reverse direction is VS-Code-only. A large fraction of working Clojure developers use Emacs/CIDER or Cursive — for them code→element is dead weight and element→code degrades to 'read the file:line off the breadcrumb and jump manually.' React DevTools works regardless of editor, so the SPA-parity claim is really parity-if-you're-on-Joyride. This trade-off is flagged as coupling but its exclusion of a big chunk of the audience is never named.
- Ch.17's dossier is enumerated more than demonstrated. Flow gets a full, superb treatment; the other six drill-downs (value, branches, source, hiccup, value-threads, last-error) get one bullet plus a one-line route comment each. 'Why is this section empty?' (branches) and 'the markup a component became' (hiccup) are each interesting enough to want a worked example, but neither is shown running — they consume conceptual space as 'variations on the same template' without paying it back.
- Some of ch.16–17 is gold-plating whose payoff is thin against the book's 'best build, ruthless' ethos. The lexical/temporal dual-tree (children/children-rt, roots/roots-rt, two reduces, the whole re-parenting apparatus) exists to power a toggle, justified only as 'the tree that reads the way a human expects' — a skeptic asks why one lexical tree with a :lazy note wouldn't do. Likewise ch.17's draggable-detach + popout-window (window.opener correlation) is competent front-end polish the chapter itself files under 'the polish,' but it dilutes the two genuinely novel ideas.
- The build-spans presentation asks a lot for uncertain reward: it is flagged as read-only-if-you-care and heavily elided with ,,, , yet the prose then walks three interacting tricks (folding, call-site recovery, lazy re-parenting) in a single let. A discerning reader is left unsure whether to study it or skip it, and the density there buys the least-justified feature (the toggle).

**Underdeveloped / raised-not-paid-off:**
- The reverse direction (code→element) never gets a concrete bug scenario the way the construction view gets the stale-title story; its daily utility is claimed but not shown.
- branches (/dev/__branches, 'why is this section empty — the SSR equivalent of the button isn't there because you're logged out') is a genuinely nice idea reduced to one sentence; it deserves the worked example flow got.
- hiccup (/dev/__hiccup, 'the markup a component became') is raised in a bullet and never demonstrated, despite being one of the more intuitive drill-downs.
- The Emacs/CIDER and Cursive audience is implicitly excluded from the editor-bridge payoff but never addressed — even a paragraph on what these users still get (the breadcrumb's file:line for manual navigation) vs. what they lose (all of code→element) would close the gap.
- Alt+wheel chain-walking (ch.15) and the icicle degree-of-interest focus (ch.17) are described in prose only; both are interaction models whose discoverability and feel a skeptical reader would want a sentence on.
- Whether a team should build this at all versus just opening FlowStorm's own UI is confronted in ch.17 but not resolved — the DOM-correlation advantage is stated without weighing it against what FlowStorm's debugger already delivers for free.

**Voice & pacing:** Register is consistent and distinctive across all three chapters — dense, confident, essayistic, heavy on em-dashes, italic emphasis, and the '> Decision / Lesson / Trade-off' callout pattern. It reads well and the voice never wobbles. Ch.15 and 16 earn their density: the argument keeps moving and the callouts mostly carry real content. Pacing sags in ch.17's middle third (the dossier enumeration and the detach/popout polish), where the prose is doing front-end tour-guiding rather than advancing an idea, and the three-chapter run is the longest single-feature stretch in the book — the cumulative load is real and will test even a sympathetic reader's patience precisely where the payoff is thinnest. The callout-and-italics machinery is worked hard enough across three chapters that it occasionally tips from emphatic into mannered, though it stays on the right side of house style.
  
**Standout:** Ch.16's 'DB ops from expressions, not from names' section, culminating in the lesson 'instrument the boundary you can name, but read the expressions you can't.' It diagnoses precisely why the readable design (badge frames whose name is on a wrapper allowlist) is structurally blind to raw d/q / d/as-of / d/history — the very time-travel reads a recipe-versioning app is built on, which are never compiled by us and so record no :fn-call — then replaces it with detecting Datomic ops by expression shape and resolving each :coord back to real Datalog-as-data. It is the rare passage that is simultaneously a correct deep-systems insight, tightly bound to the book's chosen domain, and a reusable principle stated crisply enough to travel to any codebase.

### Reload-loop+PE (18-morph-reload, 19-progressive-enhancement) — 8/10
*Two strong, tightly-voiced chapters bound by a deliberate parallel thesis — "a save is not one thing" (18) and "a rich interaction is not one thing" (19) — that mostly earn their density and are unusually honest about the costs of their own choices. Ch.18 is close to exemplary: one clear spine (the delivery matrix), every claim shown rather than asserted, and two genuinely best-in-class decision boxes. Ch.19 is more ambitious and correspondingly softer at the seams: its central "sediment" metaphor is oversold against the toolbox-drawn-from-simultaneously reality the chapter itself demonstrates, it annexes tenancy and caching into a front-end chapter in a way its own thesis warns against, and its lone rival-dismissal (LiveView) falls below the book's steelmanning standard. Publishable as-is and strong; not yet best-in-class because Ch.19's framing and scope discipline don't match Ch.18's.*

**Strengths:**
- Ch.18's 'An ES module is a URL-cached singleton' box is the standout of the cluster: it enumerates the three real options (eval, re-import under a cache-bust URL, full reload), rejects eval on CSP grounds and re-import on zombie-handler grounds, and lands on 'reload is the correct behavior, not a fallback.' This is exactly the problem→steelmanned-options→justified-choice structure the book promises, executed at full strength.
- Ch.18's 'Decision — morph <body>' box turns a subtle bug into the chapter's sharpest argument: a <main>-only morph reports total success while leaving stale nav chrome with no banner — manufacturing precisely the silent staleness the chapter builds reload-error to prevent. The reasoning is self-consistent with the rest of the chapter and names the honest residue (head edits still go stale) instead of hiding it.
- Ch.18's lifecycle-hardening section is concrete production engineering, not hand-waving: the two-writers-racing-styles.css failure mode, stop-waits-for-actual-exit before restart, the JVM shutdown hook against orphaned Tailwind, and idempotent start are each motivated by a specific failure they prevent.
- Ch.19's tenant-isolation section ('Where ownership is enforced') is the best architectural passage in the cluster: three real candidates for where the guarantee lives (middleware / per-handler / data-layer waist) each with genuine tradeoffs, choosing the narrowest waist because every read and write already passes through id→eid conversion. The 'foreign and missing are indistinguishable → a prober learns nothing, not even that an id exists' point is a sharp, correct security insight, and it's verified in source (entid-owned family at src/myapp/db/core.clj:158).
- Ch.19's 'this path does not morph' decision on drop is exactly the kind of hard-won detail that makes the book credible: the optimistic order is authoritative by construction, and morphing a view-transition-name'd row mid-flight leaves a ghost snapshot — flagged as 'a real bug we hit and backed out.' Concrete, honest, and it lands the Layer-4 demotion thesis.
- The 'Reordering is not a version' box is a small gem of domain reasoning: because every edit is a Datomic transaction, a position change would otherwise surface as a phantom empty version, so the timeline is restricted to versioned-attrs. Invisible until it bites, which is the book's house style at its best.
- Register and motif discipline across both chapters is excellent: 'narrowest correct response/mechanism' is a genuine through-line, the two theses are explicitly twinned, and both chapters close by naming what they did NOT do and what the choice costs.
- Ch.19's honesty section ('Where the layering bottoms out') refuses to oversell: drag is an enhancement not a baseline, the capability degrades but the data does not, and the platform-support table is candid about anchor-positioning and cross-document View Transitions still being uneven.

**Weaknesses:**
- Ch.19's organizing metaphor is imprecise and slightly oversold. The intro claims five layers 'each degrading cleanly to the one below it ... they are sediment,' but the chapter's own flagship example uses four layers at once (Pointer Events + CSS motion + View Transition + morph/no-JS buttons), and the actual degradation story is 'each falls back to the baseline / to a no-animation version,' not 'to the layer immediately below.' Disabling View Transitions doesn't drop you to Layer 1; it drops you to the instant mutation. The correct framing (opportunistic enhancement whose absence costs only its own capability) is also present in the same paragraph, so the chapter states both the accurate and the inaccurate version and leans on the catchier, weaker one.
- Ch.19 annexes tenancy, HTTP caching, and prerender/performance into a chapter titled 'Progressive Enhancement' — ironic given the chapter's thesis is that one powerful tool should not annex jobs it isn't the narrowest fit for. The tenant-isolation material is the most security-critical content in this cluster (arguably in the book) yet lands as a subsection of Layer 0 in a front-end chapter; the 'private vs public / wrap-no-cache-authenticated / immutable basis-t' passage is a dense six-sentence caching argument only loosely about enhancement layering. Both are excellent, but a skeptical reader will feel the definitive tenancy treatment is buried in the wrong chapter.
- The one rival that gets dismissed — LiveView/Livewire — falls below the book's own steelmanning bar. 'Pull ephemeral state back to the server and turn the browser into a dumb terminal' is the SSR-purist caricature; LiveView explicitly ships JS hooks and phx-update="ignore" for exactly the ephemeral-in-the-browser cases the chapter says it can't handle. A book whose method is steelmanning the rival engineer gives this serious rival a one-line strawman.
- Ch.19's Layer 1 reads more as a feature tour than an argument. Popover + anchor positioning, :has(), native <dialog>, @starting-style, and the stretched-link trick are each justified locally, but the 'narrowest correct mechanism' through-line that disciplines Ch.18 goes slack here — the stretched-link discussion in particular is a nice CSS technique only loosely tied to the layering thesis. Combined with the intro's own 'keep a reference tab open' admission, the middle of Ch.19 tips from earned density toward catalog.
- Ch.18's 'Decision — morph <body>' box is the one place the prose becomes a wall: a single ~300-word paragraph fuses three distinct decisions (why <main> is wrong, the head-edit residue it accepts, and the data-myapp-overlay overlay contract). Each is worth making; packing them into one block undercuts the otherwise clean pacing.

**Underdeveloped / raised-not-paid-off:**
- 'Islands' is in Ch.19's title and layer table but the term is used loosely. What's built is a Stimulus-style controller registry (connect/disconnect on data-controller markers), which is closer to progressive-enhancement controllers than to islands-architecture partial hydration. The chapter never distinguishes the two, so a reader who knows the islands literature will find the label doing more marketing than defining.
- The reuse of ignoreActiveValue as the animation-suppression signal (confirmed in dispatcher.js: it gates value-clobbering, the pending progress bar, AND View Transitions) is presented as natural, but it's one flag standing in for three semantically-distinct decisions. It happens to work because high-frequency keystroke morphs coincide on all three — but the chapter asserts the coupling without naming that it's a coincidence it's relying on.
- The four 'migrated' enhancers (live-preview forms, deferred <details>, out-of-form previews, admin stat poller) are named but never explained — a curious reader gets no sense of what a 'deferred <details>' or an 'out-of-form preview' actually does. Fine as passing examples, but they're invoked as if familiar.
- Both chapters lean heavily on forward references for load-bearing machinery: Ch.19's security claim that inline scripts are CSP-hash-gated, and the session-derived user-eid that the whole tenancy story depends on, are both deferred (to ch.24 and the auth chapters). Inherent to the book's ordering, but it means chapter 19's strongest claims are checks that only later chapters cash.

**Voice & pacing:** Register is consistent and high across both chapters — confident, essayistic, controlled colloquialisms ('blunt instrument,' 'gravy on a meal that is already complete') that never tip into cute. The deliberate parallel construction between the two chapter theses is one of the cluster's real strengths and is signposted explicitly. Density is mostly earned: Ch.18 is tight end-to-end with a clean matrix→messages→client spine and a per-edit walkthrough that pays off the abstraction. Ch.19 is where pacing strains — the middle (Layer 1 feature tour, then the caching/prerender block) packs more surface than the through-line can discipline, and the author's own 'keep a reference tab open' caution is an honest tell that several pieces are under-explained by design. The single worst pacing spot is Ch.18's morph-<body> decision paragraph (three decisions in one block). Nothing reads badly; a couple of Ch.19 stretches read fast rather than well.
  
**Standout:** Ch.18's 'An ES module is a URL-cached singleton' blockquote — it enumerates the three honest ways to pick up edited module code (eval, re-import under a fresh URL, reload), rejects eval because the book's own CSP forbids unsafe-eval, rejects re-import because it leaks zombie event listeners and setIntervals on every save, and concludes that a full reload 'is the correct behavior, not a fallback.' It is the book's problem→steelmanned-options→justified-choice method firing on all cylinders, and it retroactively justifies the entire shape of the delivery matrix.

### Auth+e2e (20-auth-tokens, 21-auth-email-flow, 22-e2e-testing) — 8/10
*A strong, unusually honest cluster that mostly practices what the book preaches: it names the costs of its own choices (single-instance rate limiting, mail-scanner nonce consumption, login CSRF, the getMagicLink ordering coupling) with a candor rare in practitioner writing, and the E2E chapter's "defend each choice against its alternative" section is exemplary. It is held back from best-in-class by three genuine editorial gaps: the opening dismissal of passwords is asserted rather than steelmanned (inconsistent with the book's stated method), the cookie-session section oversells "no server-side store" as "ideal" while silently omitting the well-known revocation cost, and — most damning — the E2E chapter never writes the six-line replay test that would prove the elaborate one-shot nonce from Ch.21 actually works end to end, leaving its single best demonstration on the table. Fix those and it is a 9.*

**Strengths:**
- Ch.21's login-CSRF section (line 380) is the best security-honesty writing in the cluster: it names a real, non-obvious gap (/auth/verify mutates on the URL token, not the cookie, so SameSite never engages; an attacker can silently sign a victim in as themselves), specifies the correct fix (bind verification to a short-lived cookie set at /auth/request), and then declines to build it with an explicit, defensible cost statement rather than pretending the gap isn't there.
- Ch.21's corporate-mail-scanner passage (line 473) turns a subtle production failure mode into a genuine insight: a strictly one-shot nonce plus Microsoft Defender's link pre-fetching means the scanner consumes the link before the human clicks. It ties this back to the progressive-enhancement chapter's refusal to prerender /auth/*, showing the same principle enforced in two places.
- Ch.20's isEqual caveat (line 165) is technically precise and I verified it against the OpenJDK source: the folding of lenA-lenB into result, walking every byte of the first argument, and the single lenB==0 early exit are all accurate, and the reuse caveat ('hash both sides to a fixed width first') is the correct generalization. This is real cryptographic literacy, not cargo-cult.
- Ch.20's get-or-create race analysis (line 259) is model engineering writing: it names the check-then-act window, admits the upsert silently rotates :user/id, states precisely why it costs nothing today (nothing reads :user/id; the session keys on email), and points at the exact future fix (a transaction function, same as the nonce CAS). It even circles back at line 437 to note what the duplicate-upsert test does NOT assert.
- Ch.22's isolation section (lines 329-331) dissects the parallel-worker race concretely — an unscoped DELETE landing between another worker's POST and its getMagicLink read, 'the kind of race that fails one run in twenty and never on your machine' — and derives the shared-nothing design from it, including the honest caveat that the per-IP rate limiter is the one truly process-global bucket.
- Ch.22's retries=0 decision box (line 453): 'A flaky E2E test is a defect, not weather.' It states the real cost it's accepting (a genuine environmental flake fails the whole build) and takes it on principle. This is the book's opinionated-with-a-named-cost method at its sharpest.
- Ch.22's 'Why the harness is shaped this way' section (459-469) genuinely steelmans each rejected alternative — Etaoin (keeps one language but loses auto-wait + lifecycle), in-test server (awkward JVM-from-Node supervision), DI over alter-var-root (machinery for isolation a separate process already provides), a Mailpit container (lifecycle weight in-memory Datomic was chosen to avoid). Each rival gets its best case before being declined.
- The getMagicLink coupling note (Ch.22 line 300) is a small masterclass in honesty: it explains why a single request.get with no retry is safe (auto-wait plus synchronous inline send), then immediately names the coupling that makes it safe and the exact change (background thread/queue) that would break it.

**Weaknesses:**
- Passwords are dismissed, not steelmanned. Ch.20 opens 'Passwords are a liability' (line 3) and asserts magic links 'sidestep all of this' with no genuine weighing of the rival: no password-manager ergonomics, no instant/offline login, no acknowledgment that magic links impose a 15-second email round-trip and a context switch on every login, and — most importantly for a security book — no admission that passwordless makes the email account a single point of failure with no second factor (line 5 even assumes the inbox is 'already secured with their own MFA,' which is often false). This is the one place in the cluster where the book's own problem->steelmanned-options->choice method is skipped; the naive/rival option gets a strawman.
- The cookie-session section oversells 'no downside.' Ch.21 line 362 calls a client-held session 'ideal' and line 368 frames 'no server-side session store' as 'not a shortcut that trades away safety.' But it never names the standard, well-known cost: no revocation. A copied cookie is valid for the full 30 days; logout only clears it client-side and cannot invalidate a stolen copy; there is no 'log out all sessions' or post-compromise kill switch. This is conspicuous because the chapter is scrupulously honest about every OTHER cost (rate-limit multi-instance, login CSRF, mail scanners) and because revocation is not on the afterword's out-of-scope list. The confident 'ideal' framing invites exactly the pushback it doesn't answer.
- The E2E chapter never tests the system's single most important security property. Ch.21 spends a full section building a CAS-based one-shot nonce ('a genuine token can be presented twice... we need a compare-and-swap'), and Ch.22 exists to verify 'stateful behavior nearly impossible to unit test.' Yet neither the book nor the repo (confirmed: e2e/auth.spec.js ships only new-user, returning-user, logout) contains a test that clicks a magic link twice and asserts the second click is rejected — nor an expired-token test. Replay enforcement is the textbook case where E2E earns its keep (HTTP + session + Datomic CAS interacting), and the book leaves its best demonstration unwritten while claiming 'nothing mocked between the click and the database.'

**Underdeveloped / raised-not-paid-off:**
- 'The interstitial' carries a lot of argumentative weight but is never built or even sketched in code. It is invoked as the fix for mail-scanner nonce consumption (line 473), as the cost-raiser for login CSRF (line 380), and as 'the natural home' for a future browser-binding fix — three separate tensions resolved by reference to one component that exists only in prose. Given how often it's leaned on, a reader wants at least a code sketch of the GET-renders-a-confirm-POST page.
- 'Scanner-tolerant nonce semantics, where consumption is bound to the session actually issued rather than to the first fetch' (line 473) is dangled as the other escape from the mail-scanner problem and then dropped. It's the more interesting of the two options and gets one clause.
- Signing-key rotation is mentioned as a feature ('change your key, all outstanding tokens become invalid immediately', Ch.20 line 393) but the operational reality is never developed: rotating the magic-link key invalidates every in-flight login link, and rotating the 16-byte SESSION_KEY would log every user out at once. For a book subtitled 'Hardened from the First Commit,' the key-rotation runbook is a natural expectation that goes unaddressed.
- Corporate-mail-filter prevalence is underweighted relative to the book's 'production SaaS' ambition. Ch.21 honestly names the failure but waves it off with 'a recipe box is a low-value target' — yet Microsoft 365/Defender pre-fetching is ubiquitous in exactly the B2B market a 'SaaS from scratch' might target, meaning for those users EVERY first click fails. The tension between the demo domain (personal-email recipe box) and the stated ambition is real and only half-acknowledged.

**Voice & pacing:** Register is consistent, dense, and confident across all three chapters, and it earns its density almost everywhere — the prose is doing analytic work, not padding. Ch.22 reads the most smoothly and its 'almost anticlimactic' framing of the specs is well-earned by the harness build that precedes it. The one passage that flirts with over-investment is Ch.20's isEqual caveat block (line 165): it is accurate and genuinely educational, but it is a very long, JDK-internals-deep digression about a helper the app already uses correctly, and a skeptical reader may read it as a flex rather than a necessity. Nothing reads as filler; if anything the failure mode is occasional over-elaboration, which is the book's identity rather than a defect.
  
**Standout:** Ch.21, line 322: the construction-view tie-in — 'The generic page tells the visitor nothing; the recording tells you everything; the asymmetry is the point. The two halves of the design -- opaque outside, transparent inside -- are the same decision made twice.' It fuses the security requirement (never reveal which auth gate failed) with the book's construction-view thesis (total internal observability) into a single principle, and does it in three sentences. It's the moment the cluster stops being a competent auth tutorial and becomes an argument with a spine.

### Admin+assets (23-admin-dashboard, 24-asset-pipeline) — 8.5/10
*Two strong chapters that mostly earn their density, anchored by genuinely excellent security writing in Ch.24. Ch.24 does real options analysis (hash-vs-nonce CSP), threat-models its one concession instead of hand-waving it, and shows spec-level precision (COOP/CORP/COEP) that distinguishes a serious book from a competent one. Ch.23 is solid and honest but softer: its central "disposable analytics DB" justification is told rather than shown, it over-repeats its thesis, and a seam opens between its "self-contained" middleware listing and the repo-shaped handler it feeds. The cluster is publishable as-is, with Ch.24 pushing toward best-in-class.*

**Strengths:**
- Ch.24 CSP hash-vs-nonce: real options analysis, both steelmanned. It states when a nonce is the right tool (third-party embeds, a CMS you don't control — a per-request wildcard when you cannot enumerate) and then earns the hash choice on the enumerability the app actually has, naming the price honestly ('a new inline script becomes a code change') and reframing it as the feature.
- Ch.24 style-src 'unsafe-inline' concession is threat-modeled, not excused. It names the residual risk (CSS injection -> clickjacking/phishing), walks the textbook attribute-selector CSRF-token exfiltration escalation, then shows why img-src 'self' data: and font-src 'self' under default-src 'none' close the exfil fetch — and that script-src carries no 'unsafe-inline' so it still can't run script. This is the rare book that quantifies a compromise's blast radius.
- Ch.24 COOP/CORP passage explains not just what each header does but what the app deliberately does NOT set (COEP/require-corp) and why — 'we need none of what it unlocks.' Precision about the absence is as valuable as the presence.
- Ch.23 case-insensitive admin-email check: steelmans the strict-correctness position (RFC 5321 makes local parts case-sensitive) before overriding it on operational grounds, and frames the failure concretely — one stray capital locks out the single person the gate exists to admit.
- Ch.23 JSON-vs-HTML branch is argued from a concrete failure: a fetch silently following a 302 to an HTML login page and handing the poller HTML to parse as JSON. That is exactly the kind of failure that reads as a mysterious client bug, and naming it justifies the :json? split.
- Ch.23 live-polling lifecycle names two DISTINCT naive failures (import-time IIFE leaks an interval across morph nav forever; per-morph wiring stacks without teardown) and resolves both with one refcounted interval on the controller registry. Genuinely the strongest technical argument in Ch.23.
- Ch.24 shows the regression tests that pin its claims (escaping, strict CSP, importmap authorization) rather than asserting they exist — the csp-is-strict test's negative assertions ('must NEVER allow unsafe-inline/unsafe-eval') are the right shape for a hardening guarantee.
- Both chapters flag their own asides honestly: the @property counter section is marked skippable for a security reader, and the import-map integrity caveat openly warns that map-level integrity has uneven browser support and should be treated as defense-in-depth, not the sole guarantee.

**Weaknesses:**
- Ch.23's 'disposable analytics database' payoff is asserted, never shown. The separation is justified by being able to 'delete and recreate without touching user data,' but the chapter never gives a concrete scenario in which a solo operator would actually wipe analytics — and then spends a full paragraph defending against the nonce-replay window that the disposability itself creates. Net: the reader is asked to defend a real coupling for the sake of a benefit that is never motivated. Ground the disposability in one real reason to wipe (schema reset, GDPR bulk purge, corrupted event stream) or the separation reads as a rationalization.
- Ch.23's 'self-contained' claim frays at a seam. The shown wrap-admin reads [:session :user-email] but never stamps :user-email onto the request top level; yet the handler listing reads (:user-email request) — which is the REPO form (verified: routes.clj wrap-current-user stamps :user-email/:admin?). A reader building strictly from the chapter's standalone middleware gets a nil email in the header. The 'How the repo factors this' note flags the divergence in principle, but mixing the standalone middleware with the repo-shaped handler in the same chapter quietly undercuts the 'minimal thing that works on its own' promise.
- Ch.23 over-repeats its thesis. 'Authorization is the part that has to be right; everything downstream is conventional query-and-render' appears in the opening framing, again mid-chapter, and twice more in the closing two paragraphs. Once is a good frame; four times reads as padding in a chapter that is otherwise economical.
- Ch.24's vendored-library exemption has a blind spot the chapter is proud of closing everywhere else. The whole thesis is verify-assets proving 'a name can never lie about its contents' — but the one file exempt from content hashing (idiomorph-0.7.4.min.js) rests on the unenforced convention that 'the version string already changes whenever the bytes do.' A locally patched vendored file with an unchanged version string plus the one-year immutable cache is exactly the lying-name failure the gate exists to prevent, and nothing catches it. The chapter verifies the vendored file's existence but not its byte-integrity, and never acknowledges that the one hole its gate can't close is the one file it exempts.

**Underdeveloped / raised-not-paid-off:**
- Ch.23's JVM memory cards (used/free/total/max, polled every 20s) are rendered and wired but never justified: why heap belongs on an admin dashboard, what a solo operator actually does with it, and how it relates to the richer signals it omits (GC pressure, request latency, error rates). The chapter half-concedes the counters are 'polish'; the JVM readout gets no such honesty and sits as unexamined vanity metrics next to a closing line that boasts 'no monitoring service to pay for.'
- Ch.23's 'skip this, it's just polish' framing of the animated counter undersells that the polish shaped the core rendering. Because the value must animate, it rides on a --stat-value custom property read by CSS counter() instead of being plain textContent — a measurably more complex data path than a text node the poller sets directly. The section is self-aware about being decorative but doesn't own that the decoration drove the stat-card's whole attribute design.
- Ch.24's 'cannot drift' claim glosses the minifier as the one transform that runs only in prod. Dev serves JS unminified; prod serves esbuild-minified; so 'differ only in how the same files are delivered — not in what they are' is slightly overstated for JS, and esbuild-minify is the one unexamined behavior vector between dev and prod. The chapter's own intro is more careful ('the JavaScript differing only by minification'); the 'one engine' section should carry that same precision rather than eliding it.
- Ch.24's birthday-bound rigor is welcome but models the wrong collision. It bounds accidental collisions across ~65k distinct files, whereas the collision that actually causes a stale serve is version-over-version at a single logical name (a ~2^-32 single comparison per deploy). The margin conclusion still holds and the rigor is a net strength, but the model is looser than the confident prose implies.

**Voice & pacing:** Register holds across both: essayistic, dense, opinionated, and consistently honest about limits — the house style. Ch.24 sustains a very high information density (spec-level CSP/SRI/import-map detail) and earns nearly all of it; its security sections are the cluster's density high-water mark and still read cleanly. Ch.23 is looser — the same points recur (the authorization-is-the-only-hard-part refrain, the 'zero JS' boast) and it would tighten by 10-15%. Both chapters are commendably self-aware about their asides (counters flagged skippable, integrity caveat flagged uneven), which keeps the density from feeling like showing off. The prose rarely asserts where it should show, with the notable exception of Ch.23's disposable-DB motivation.
  
**Standout:** Ch.24's paired treatment of the CSP: the hash-vs-nonce decision (both mechanisms steelmanned, the hash chosen on enumerability, the price named and reframed) immediately followed by the style-src 'unsafe-inline' concession threat-modeled down to the attribute-selector CSRF-token exfiltration and shown to be largely closed by img-src 'self' data: / font-src 'self' under default-src 'none'. That sequence — justify the strict choice, then refuse to hand-wave the one place you couldn't be strict, then quantify exactly how much risk the concession actually leaves — is precisely how a serious security chapter should read, and most books get neither half right.

### Prod-hardening+close (25-lighthouse, 26-ci-cd, afterword) — 8/10
*A strong, largely convincing close to the book, carried by a caliber of operational honesty that most production-engineering writing never reaches: the chapters repeatedly name the sharp edges of their own designs rather than selling past them. Ch.25 earns its density; the afterword is clean and lands the wager. Ch.26 is the weakest of the three — not because it is wrong but because its back half re-treads its own annotated YAML, and a couple of its central quantitative choices are stated rather than fully closed. Publishable as-is; a targeted pass on ch.26's redundancy and two under-argued numbers would push it toward best-in-class.*

**Strengths:**
- Ch.26's 'What the push-only guard does not buy' callout (line 498) is the standout of the whole cluster: it identifies that a pull request runs untrusted code as root on the self-hosted runner where the deploy key sits as a plain file, so a PR can exfiltrate production SSH access without any deploy step — names it as 'the larger of the two trust holes' and gives three concrete fixes. This is a real, commonly-missed self-hosted-CI vulnerability that most books would never surface.
- Ch.25's landmark honesty (line 278) is exemplary intellectual integrity: the author actually checked that Lighthouse 12.x scores `landmark-one-main` at weight 0, admits a page missing `<main>`/`<nav>` still scores a clean 1.0, and then gives the *real* reason to keep them (assistive tech + `<main>` being the morph target the dispatcher enforces). A lesser book claims Lighthouse enforces landmarks; this one disproves that and keeps them anyway on better grounds.
- Ch.25's 'honest limit of seeding the minimum' (line 88) pre-empts the exact objection a skeptical engineer raises: a 100 on one-row pages proves the render path, not that it survives production data sizes — with the correct prescription to seed near the realistic ceiling for cost-scaling pages.
- The auto-auth server is named plainly as 'an unauthenticated admin bypass by construction' (line 67) with the loopback-only, test-classpath-only boundary stated as an imperative, not a footnote.
- The ch.26 deploy script (lines 508-547) teaches genuinely valuable ops knowledge: that `systemctl start` returns when the process is *spawned*, not *serving*, so a jar that boots-and-dies passes a naive `is-active` check and surfaces only as a silent crash-loop — closed by polling the real `/health` endpoint, with a least-privilege three-sudo-rights grant.
- Ch.26's layer-caching insight (line 108) is subtle and correct where most get it wrong: a `curl .../releases/latest` line does NOT re-fetch on an unchanged Dockerfile because Podman reuses the cached RUN layer byte-for-byte, so 'latest'-tracking tools are effectively frozen until the cache is busted.
- Genuinely steelmanned micro-options: font-display `swap` vs `optional` (line 266, naming swap's late-CLS cost), self-hosted vs hosted CI with the actual GitHub-Actions port diff spelled out (line 15), and Podman vs Docker ('a team standardized on Docker would reasonably keep it').
- The 'contract, not an achievement' frame (ch.25 line 9) is introduced, threaded through every fix, and paid off in the conclusion — the chapter's spine holds.

**Weaknesses:**
- Ch.26's back half is genuinely redundant, not merely dense. The manifest-must-travel-with-the-assets point is made three times (lines ~399, 502, 549); the immutable-cache / verify-assets 'last chance before a year-long cache poisoning' argument appears at 407 and again at 549; the CACHE_VOLS / `:Z` SELinux mechanics and the layer-cache-freezes-tools point are each explained twice (inline in the image/env sections, then re-explained in 'Stage by stage'). The 'stage by stage' section largely re-narrates the already-annotated YAML above it. This is repetition, which is a different thing from the book's identity-density.
- The `numberOfRuns: 1` + `0.95` floor choice slightly undercuts itself on the chapter's own numbers. The text says performance jitter 'move[s] it by a few points' and the representative run is 98 — leaving only ~3 points of headroom above a 95 floor, with a single observation (the highest-variance case). A single bad run could dip below the floor and produce a spurious red, the exact thing a gate should avoid. The chapter frames raising to 3 runs as 'if you'd rather hold performance at a hard 100' — but medianing's primary payoff is variance reduction / false-fail protection at *any* threshold, which the framing omits. The tradeoff is presented as speed-vs-strictness when it is also speed-vs-flakiness.
- Ch.25 never states the load-bearing caveat that a Lighthouse accessibility 100 is not WCAG conformance — automated auditing catches only a minority of real a11y issues. The chapter's entire thesis is that the 100 is a contract, so the one thing it owes the reader is what the contract does *not* cover. The landmark paragraph gestures at this ('assistive technology navigates by them whether or not Lighthouse grades them') but never says the plain sentence: 'a11y 100 here is a floor, not a certificate.' Given how honest the rest of the chapter is, this omission stands out.
- Ch.26 installs babashka in the CI image (line 73) but it is used nowhere the book shows — verified: both `./reformat` and `./lint` are bash, and the pipeline calls `clj-kondo` directly (line 375 keeps `./lint` local). Every other tool in the Dockerfile gets its role explained; babashka just appears, and a reader scanning the image asks 'where is bb used?' and gets no answer.

**Underdeveloped / raised-not-paid-off:**
- The deploy script — 'the single highest-stakes line in the whole pipeline' by the book's own words — is the one listing explicitly NOT in the companion repo ('the shape it takes'). The book's credibility rests on 'every listing is lifted from working code,' so the highest-stakes component being the one un-exercised, illustrative piece is an honest tension the chapter names in passing but never squarely owns.
- The 50% coverage floor (verified in deps.edn) sits unexamined next to a wall of hard-100s and strict-everything. Within this cluster it reads as a conspicuously soft number, and the chapter states it ('the same 50% coverage floor') without one sentence on why 50 is the right bar when everything else is pinned at perfection. A skeptical reader's 'why so low, given the rest?' goes unanswered here.
- The afterword's 'Swap the database and see which wrappers the java.time bridge was really buying you' (line 25) is glib given the book spent a chapter arguing the recipe-versioning domain reads point-in-time views and history *straight out of Datomic's* time model. Swapping the DB would detonate the domain design, not just exercise a bridge — the invitation is much harder than the sentence implies, and the tension with the Datomic-as-load-bearing-substrate argument goes unacknowledged.
- Forgejo is chosen almost entirely on 'you already self-host the forge,' but rival self-hosted forges/runners (Woodpecker, Drone, GitLab CI) are never weighed — the only steelmanned alternative is hosted GitHub Actions. For a book built on problem→options→choice, the forge decision is under-optioned.
- The afterword's claim that adding a feature end-to-end means 'one place for each, because there was one source of truth all along' (line 25) idealizes an architecture that in practice touches schema + handler + view + test + i18n strings + possibly CSP hashes for one feature. A reader who actually built the app knows it is more than 'one place for each'; the rhetoric slightly overstates.

**Voice & pacing:** Register holds and matches the book's dense, essayistic, opinionated voice; the prose is controlled and the technical exposition is precise throughout. Ch.25 and the afterword read cleanly at a good clip. Ch.26 is the pacing outlier: it leans heavily on stacked blockquote callouts — five or six of them — and while each is individually excellent, the alternation of inline prose, boxed aside, prose, boxed aside fragments the read, and the 'stage by stage' section re-treads material the annotated YAML already covered. The afterword's closing paragraph (line 27) tips slightly into the grandiose and recycles phrases ('the system answers' appears twice within two paragraphs; 'when your constraints differ from ours' echoes the earlier scope section) — the high register is mostly earned but the very last sentences pile up. Net: excellent sentence-level craft; ch.26 specifically would read better ~15% shorter with the duplicated deploy/asset-integrity arguments consolidated.
  
**Standout:** Ch.26, the 'What the push-only guard does not buy' callout (line 498): the recognition that `if: github.event_name == 'push'` stops a PR from *deploying* but not from *running untrusted code as root on the runner*, where the deploy key sits as a plain file it can copy out and walk away with production access — named as the larger of the pipeline's two trust holes, with real (non-trivial) fixes. It is exactly the kind of sharp, self-incriminating operational truth the genre usually omits.

## 8. DO NOT TOUCH — verified non-defects (standing guard list)

These look wrong and get re-flagged by fresh reviewers; each is verified correct against primary sources. A naive "fix" introduces an error. Re-read before changing:

1. **Ch.08** — "Datomic binaries ship from Maven Central under Apache 2.0, though the source stays proprietary." Correct: free binaries ≠ open source. (Official 2023 announcement + POM.)
2. **Ch.08** — "`d/q`, `d/pull`, `d/entity` all return `java.util.Date`." Correct: raw Peer returns `Date`; the wrapper converts to `Instant`. The docstring's "either/or" is the wrapper-vs-raw distinction, not a contradiction.
3. **Ch.17** — the `:ambiguous (render reordering)` label faithfully matches the shipped tool's own UI string; changing the prose alone would desync it from the product. Product decision, not a doc fix.
4. **Ch.16 vs Ch.17** — the illustrative basis-t (`@t1007` vs `@t1031`) is cosmetic and deliberately deferred (the "right" value is semantically subtle).

Also **not gaps** (honestly conceded as deliberate scope in the afterword): billing/subscriptions; horizontal scale / multi-node / read scaling; observability beyond stdout + CSP reports; backup-and-restore; background jobs / scheduled work; a real outbound-email service; offline-first sync / real-time collaboration / high-frequency direct manipulation.
